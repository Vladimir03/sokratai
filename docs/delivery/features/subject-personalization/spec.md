# Персонализация по предметам: профиль репетитора → весь кабинет (+ ученик)

Статус: **СПЕКА УТВЕРЖДЕНА владельцем (решения 2026-07-23), реализация не начата.**
Решения владельца зафиксированы через AskUserQuestion 2026-07-23; карта текущего
состояния — разведка кода той же датой.

## Section 0: Job Context

- **Core Job (репетитор):** «Настроить рабочее место под мой предмет без трения,
  чтобы каждая поверхность (База, ДЗ, пробники, ученики, AI) говорила на языке
  моего предмета» — усиливает wedge (AI-проверка ДЗ): меньше шагов до первой
  ценности, меньше ошибок класса «обществознание получило физику» (репорт Милады
  2026-07-22: физ-заголовок пробника, физ-справочник, физ-дефолты).
- **Core Job (ученик):** «Получать объяснения на моём уровне (класс, цель,
  экзамен), а не универсальные» — удержание и качество AI-диалога.
- **Триггер:** в проде репетиторы 4+ предметов (Егор/физика, Милада/общество,
  Эмилия/французский, Светлана/математика) — physics-дефолты стали ошибками.

## Решения владельца (2026-07-23, locked)

1. **Сбор предметов:** онбординг-шаг после регистрации + блокирующие мини-диалоги
   на 4 поверхностях (База / создание ДЗ / добавление учеников / пробники) для
   существующих без предметов. Не только мягкие баннеры.
2. **Списки предметов:** «умные» — предметы профиля наверху и предвыбраны,
   остальные под разделителем «Другие предметы». НИЧЕГО не прячем (разовый ученик
   по новому предмету всегда возможен).
3. **Сигналы v1 (все четыре + промпты):** экзамен-фокус (ЕГЭ/ОГЭ/школа/олимпиады)
   в профиле; предмет ученика всегда; прятать/помечать нерелевантное
   (физика-only фичи); класс ученика виден репетитору. **Плюс: предмет, класс и
   экзамен-фокус — в AI-промпты ДЗ, пробников и student AI-чата.**
4. **Ученик — гибрид:** предметный контекст наследуется от репетитора
   (tutor_students.subject / per-ДЗ subject), но ученик может дополнить свои
   предметы в профиле (для самостоятельного AI-чата/практики).

## Текущее состояние (инвентарь, 2026-07-23)

**Уже есть (фундамент):**
- `tutors.subjects TEXT[]` — канонические id (`src/types/homework.ts::SUBJECTS`,
  14 шт.); пишется ТОЛЬКО из профиля (`SubjectsMultiSelect` в `TutorProfile`) и
  `SubjectsNudgeBanner` (единственный монтаж — `KnowledgeBasePage`, мягкий,
  dismiss в localStorage).
- `resolveTutorDefaultSubject(profileSubjects, lastUsed)`
  (`src/lib/tutorSubjects.ts`) — single source: ровно один контент-предмет → он
  ВСЕГДА; иначе last-used → профиль → physics. Подключён (prefill-only): формы
  Базы (`CreateTaskModal`/`TaskClassificationFields`), AI-загрузчик (`InputStage`),
  конструктор ДЗ (`TutorHomeworkCreate` L0), `HWDrawer`, `PublishFolderModal`,
  витрина Каталога (pills = union якорных ∪ тем ∪ профиля), `TutorHome` →
  `DemoCheckCard`.
- Ученик: онбординг (`OnboardingModal` + `get_student_onboarding_context`,
  миграция `20260720160000`) уже собирает класс (1–11/взрослый), цель,
  `difficult_subject` (предмет СКРЫВАЕТСЯ при `has_tutor` — наследование уже
  частично живёт); хранится в `profiles.{grade, learner_type, learning_goal,
  difficult_subject, onboarding_completed}`.
- AI-промпты ДЗ subject-aware по всем 3 путям + rubric-слой (rule 40); пробники
  subject-aware (2026-07-22).

**Гэпы (что закрывает эта спека):**
- Регистрация предметы НЕ сохраняет: `RegisterTutor` не спрашивает;
  `TutorSignupTrial` спрашивает УРЕЗАННЫМ списком, но ответ умирает в
  `user_metadata.subject` — в `tutors.subjects` не попадает (edge
  `assign-tutor-role` вставляет tutors без subjects).
- `TutorMockExamVariantEditor` — хардкод `'physics'`, профиль игнорирует.
- `TutorHomeworkTemplates` / `HWTemplatePicker` — фильтр предметов дефолтится в
  'all', профиль игнорирует.
- `AddStudentDialog` — префилл предмета ученика только при РОВНО одном предмете
  профиля; мульти-предметник получает пустое поле.
- Списки предметов везде плоские 14 пунктов без группировки.
- Экзамен-фокуса в профиле нет вообще; класс ученика репетитору не показывается;
  student AI-чат (`Chat.tsx`) не знает ни предмет, ни класс, ни цель.

## Архитектура v1

### Ф1. Сбор предметов (решение 1)

- **Схема:** ничего нового — `tutors.subjects` уже есть.
- **Новые репетиторы:** `TutorSignupTrial` — заменить урезанный select на полный
  `SubjectsMultiSelect` (мультивыбор), писать массив в `user_metadata.subjects_intent`;
  финализация (`email-verify::assignTutorRoleIfNeeded` / oauth-callback — там, где
  создаётся строка tutors) персистит его в `tutors.subjects` (зеркало `trial_intent`).
  `RegisterTutor` — НЕ раздувать форму: предметы соберёт гейт кабинета (ниже) при
  первом входе (email-confirm разрывает флоу — экран ПОСЛЕ входа надёжнее).
- **Гейт кабинета (новые без metadata + все существующие):** новый
  `SubjectsGateDialog` (реюз `SubjectsMultiSelect` + `useUpsertTutorProfile`,
  паттерн `SubjectsNudgeBanner`) — модальный диалог «Какие предметы вы ведёте?»
  при входе на 4 поверхности: `/tutor/knowledge*`, `/tutor/homework/create`,
  `/tutor/students` (+AddStudentDialog), `/tutor/mock-exams*`. Условие показа:
  `tutorProfile.subjects` пуст (контент-предметов 0). Кнопка «Позже» закрывает и
  пускает (не тюрьма — rule «не блокировать работу»), но диалог вернётся при
  следующем входе на поверхность (НЕ localStorage-dismiss навсегда, в отличие от
  баннера). Сохранение ≥1 контент-предмета гасит его навсегда (данные, не флаг).
- `SubjectsNudgeBanner` в Базе после этого можно удалить (диалог его заменяет) —
  решить при имплементации.

### Ф2. Умные списки (решение 2)

- **Новый shared `SubjectSelect`** (`src/components/tutor/SubjectSelect.tsx`):
  нативный `<select>` (16px, touch-manipulation — rule 80) с `<optgroup
  label="Ваши предметы">` (порядок канонический) + `<optgroup label="Другие
  предметы">`. Пропсы: `value/onChange/profileSubjects/includeLegacyOption?`.
  Один контент-предмет → предвыбор уже даёт `resolveTutorDefaultSubject` — не
  дублировать.
- **Мигрировать поверхности:** `TaskClassificationFields` (Create/EditTaskModal),
  `InputStage` (загрузчик), `TutorHomeworkCreate` L0, `HWDrawer`,
  `AddStudentDialog`, `TutorMockExamVariantEditor` (+ дефолт через
  `resolveTutorDefaultSubject`, убрать хардкод physics), `DemoCheckSheet`.
  Фильтры `TutorHomeworkTemplates`/`HWTemplatePicker`: дефолт = предмет профиля
  при одном контент-предмете, иначе 'all'.
- **НЕ трогать:** `HWTaskCard` (high-risk, rule 40) — он предметный селект не
  рендерит; студенческий `OnboardingModal` (свой контекст, полный список ок).

### Ф3. Экзамен-фокус в профиле (решение 3)

- **Схема:** `tutors.exam_focus TEXT[] DEFAULT '{}'` (additive миграция;
  значения `ege|oge|school|olympiad`; пер-предметная детализация — v2, не
  усложнять). Чипы в профиле рядом с предметами + шаг в `SubjectsGateDialog`
  (вторая строка чипов, опционально).
- **Потребители (prefill-only, зеркало предметного резолвера):**
  `resolveTutorDefaultExam(examFocus, lastUsed)` в `tutorSubjects.ts` — ровно
  один фокус → он; иначе last-used → первый из профиля → null. Подключить:
  exam-фильтр витрины Базы, «Тип» в `TaskClassificationFields`/каскаде
  загрузчика, `exam_type` конструктора ДЗ (L1) и редактора вариантов пробника.

### Ф4. Предмет ученика всегда + класс (решение 3)

- `AddStudentDialog`: предвыбор предмета из `resolveTutorDefaultSubject`
  ВСЕГДА (не только при одном предмете); мульти-предметник видит умный список
  (Ф2) со своими наверху. Поле остаётся редактируемым; bulk-add получает общий
  селектор предмета на пачку.
- **Класс ученика репетитору:** `profiles.grade` привязанных учеников — в
  выдачу списка учеников (существующие эндпоинты/SELECT tutor_students + join
  profiles: анти-leak чек — grade безопасен) → бейдж «9 класс» на `StudentCard`
  и в шапке `TutorStudentProfile`. Дефолт-подсказка: grade ≤ 9 → ОГЭ-контекст
  (мягкая, не переопределяет exam-фокус).

### Ф5. AI-промпты: предмет + класс + экзамен-фокус (решение 3, доп. владельца)

- **ДЗ (guided check/hint/chat):** subject + exam_type уже едут (rule 40).
  Добавить `studentGrade`/`learnerType` (из `profiles.grade/learner_type` по
  student_id — сервер уже резолвит имя ученика тем же путём) → строка в
  системный промпт: «Ученик — N класс» / «взрослый» → тон и глубина объяснений.
  Все 3 пути + bootstrap; SELECT'ы расширять точечно (whitelist, анти-leak — 
  grade student-safe).
- **Student AI-чат (`chat/index.ts`, общий путь):** сервер по userId читает
  `profiles.{grade, learner_type, learning_goal, difficult_subject}` → блок
  персонализации в системный промпт («9 класс, готовится к ОГЭ по
  обществознанию»). Клиент ничего не шлёт (anti-tamper, зеркало guided-пути).
- **Пробники (Part 2 grader):** subject/exam_type уже subject-aware; класс НЕ
  добавляем (экзаменационная строгость важнее возрастной адаптации) — решение
  зафиксировано, пересматривать отдельно.
- Инвариант: новые поля в промпт — через существующие builder'ы
  (`guided_ai.ts`, `resolveSubjectRubric`), НЕ новые пути; PII-политика: класс и
  цель — ок (имя уже едет), телефон/email — никогда.

### Ф6. Прятать/помечать нерелевантное (решение 3)

- Тренажёр формул: входов из кабинета НЕТ (только прямой URL/маркетинг) — прятать
  нечего; при будущем добавлении нав-пункта гейтить `physics ∈ subjects`.
- `DemoCheckSheet`: сэмплы есть только physics/maths/social — для остальных
  предметов честный генерик-сэмпл или подпись «пример по физике» вместо тихого
  фолбэка; «Своя задача» уже предметна.
- Карты ФИПИ (авто-балл/режимы) есть только у физики и обществознания — при
  предмете без карты honest-подпись в загрузчике/редакторе «балл вручную —
  карта ФИПИ для {предмет} не заведена» (не тихий null).

### Ф7. Ученик — гибрид (решение 4)

- Наследование уже работает (`OnboardingModal` скрывает выбор предмета при
  `has_tutor`; per-ДЗ subject ведёт AI). Добавить: секция «Мои предметы» в
  `/profile` ученика (редактируемая; хранение — расширить `difficult_subject`
  до массива `profiles.subjects TEXT[]`? — РЕШИТЬ при имплементации: v1 может
  остаться один `difficult_subject` + класс/цель, массив — если появится
  мультипредметная самостоятельная практика).

## Порядок реализации (рекомендация)

1. Ф1 (сбор) + Ф2 (умные списки) — ядро, разблокирует остальное.
2. Ф4 (предмет ученика + класс) + Ф5 (AI-промпты) — прямая ценность в диалогах.
3. Ф3 (экзамен-фокус — миграция) + Ф6 (пометки) + Ф7 (профиль ученика).
Каждая фаза — отдельный коммит с ревью; миграции additive; deploy-порядок
обычный (миграция → edge → deploy-sokratai).

## Дальнейшие сигналы персонализации обучения (v2+, ответ на открытый вопрос владельца)

Приоритезировать по мере пилота; каждый — отдельная спека:
1. **Слабые темы из данных** — у нас уже есть heatmap ДЗ + результаты пробников
   по КИМ: авто-подборка «подтянуть» (задачи Базы по западающим КИМ/темам) —
   сильнейший сигнал, данные уже копятся.
2. **Дата экзамена** ученика → обратный отсчёт + темп плана (сколько тем/недель
   осталось) в кабинетах обоих.
3. **Диагностика на входе** — первый пробник как placement: стартовый уровень →
   цель (`target_score` уже есть в tutor_students) → прогресс к цели (R2 уже
   считает `pct_to_goal`).
4. **Стиль объяснений** — за class/learner_type: краткость, строгость, примеры
   «из жизни» для 5–8 классов; у языков уже есть CEFR-механика — образец.
5. **Регулярность/темп** — стрики и напоминания уже частично есть (reminder,
   тренажёр-стрик) — связать с целью.
6. **Интересы ученика** (футбол/игры/музыка) → контекст примеров в AI-объяснениях
   — дёшево через профиль, силён для мотивации младших.
7. **Родительский контур** — отчёт родителю уже есть; персонализация = язык
   отчёта от цели/класса.

## Инварианты (при имплементации не нарушать)

- Канонические id предметов (`SUBJECTS`); `'other'` — НЕ контент-предмет;
  legacy-нормализация только через `tutorSubjects.ts`.
- Профиль влияет на ДЕФОЛТЫ и ГРУППИРОВКУ, но никогда не прячет предметы
  (решение 2) — кроме явных physics-only фич (Ф6).
- `resolveTutorDefaultSubject` — единственный резолвер дефолта; новые поверхности
  подключаются к нему, не изобретают свой.
- FK-дрейф: `tutors.user_id ↔ tutors.id` (rule 40) при любых join'ах.
- Гейт-диалог НИКОГДА не блокирует навсегда («Позже» всегда есть) и гаснет
  данными (subjects непусты), не localStorage-флагом.
- Student-safe поля из profiles: grade/learner_type/learning_goal — ок;
  ничего лишнего в student-facing SELECT'ы (анти-leak ревью на каждый).
