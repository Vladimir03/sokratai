## 0. Job Context

| Участник | Core Job | Sub-job |
|---|---|---|
| Репетитор (B2B) | **R4** — контроль качества при масштабировании | R4-операционка: вести учёт проведённых занятий + оплат без рутины |

**Запрос (дословно):** «Если занятие стоит в расписании и время прошло — может, считать его проведённым по умолчанию? Это 90% случаев. А отменить всегда можно». Уточнение по группам: «галочки был/не был на каждого ученика + поле стоимости (по умолчанию цена занятия, редактируется); если ученика не было — часто беру половину».

**Wedge:** снижает трение ключевой операционной рутины репетитора (отметка проведённого = и статус, и создание оплаты). Усиливает удержание пилота (Egor/Vadim/Elena).

**Pilot impact:** меньше кликов на рутину; точка входа в материалы занятия (связка с «Занятиями»).

## 1. Summary

Презумпция «проведено» для прошедших занятий в расписании: при заходе в `/tutor/schedule` — баннер «Прошедшие занятия (N) — подтвердите», открывающий sheet с редактируемыми суммами (и посещаемостью по участникам для групп). **«Подтвердить все»** в один тап создаёт оплаты (pending) + помечает занятия проведёнными. Ничего денежного/исходящего не происходит молча — только на подтверждении.

## 2. Problem

«Отметить проведённым» сейчас — это не просто статус: `complete_lesson_and_create_payment` ставит `status='completed'` **и создаёт `tutor_payments`** (которые питают «должников» в `/pay`-боте и payment-reminder). Репетитор кликает по каждому занятию вручную. Боль — повторяющаяся рутина на 90% занятий, которые точно состоялись.

## 3. Solution + ключевые решения (залочены)

Презумпция + подтверждение. **Деньги и любой outbound — только на «Подтвердить», никогда молча.**

1. **Оплата создаётся на «Подтвердить»** (с дефолтными, но редактируемыми суммами), не молча при наступлении времени.
2. **Презумпция + лёгкое подтверждение:** баннер «N прошедших — подтвердить» + «Подтвердить все» + исключения.
3. **Жёсткий инвариант:** никаких outbound (родителю/ученику) до подтверждения.
4. **Триггер:** конец занятия + буфер **3ч**, дайджестом при заходе в Расписание; **без cron**.
5. **Дефолт для всех** туторов (гейт подтверждения делает безопасным; per-tutor тумблер не нужен).
6. **Backlog 14 дней;** откат = атомарно сторнирует оплату (идемпотентность `(lesson_id, tutor_student_id)`).
7. **Скоуп — `regular`** (индивид + группа); trial/mock_exam/consultation исключены.
8. **Связка:** после подтверждения — «Приложить записи?» → `LessonMaterialsDrawer` (rule 98).

**Группы:** по каждому участнику `[✓ был]` + поле суммы (дефолт = цена; снял «был» → **0**, редактируется вверх). «Был/не был» — UI-контрол дефолта суммы (отдельный attended-флаг в v1 НЕ персистится).

**Индивидуальное:** `[проведено]` + `[сумма editable]` + `[не состоялось]` (без был/не был).

## 4. User Story

> Когда я открываю расписание и вижу прошедшие занятия, я хочу одним тапом подтвердить, что они проведены (поправив суммы/посещаемость у исключений), чтобы не кликать по каждому — но при этом не создать ошибочную оплату за урок, которого не было.

## 5. Technical Design

### 5.1 Поток
- **Список (клиент, без нового read):** дерив из `useTutorLessons`: `status='booked'` AND `lesson_type='regular'` AND `(start_at + duration_min мин) + 3ч < now` AND `start_at > now − 14д`.
- **Группировка строк:** есть `tutor_lesson_participants` → групповая строка (per-participant); иначе индивидуальная. (Unified-группа = `tutor_lessons.student_id IS NULL` + участники в junction; legacy-группа = отдельные per-student строки — трактуются как индивидуальные.)
- **«Подтвердить все»** → backend атомарно: `status='completed'` + `tutor_payments` (pending) на указанных суммах (`amount ≤ 0` → платёж не создаётся). «Не состоялось» → `cancelLesson` (cancelled, без оплаты).
- **После:** non-blocking «Приложить записи?» → `LessonMaterialsDrawer`.

### 5.2 Backend (CC-A)
- Атомарное подтверждение занятия с суммами (индивид: amount; группа: per-participant `payment_amount` + только >0 → payment). **Reuse/extend `complete_lesson_and_create_payment`** (group-aware). Bulk-обёртка для «Подтвердить все» (один вызов, per-lesson результат, каждое занятие атомарно). Ownership `tutor_lessons.tutor_id → tutors.id`; только `booked` + `regular`. Идемпотентность `(lesson_id, tutor_student_id)`. **RPC vs extend — решить в plan mode.** Ошибки flat `{error,code}` (rule 97), если edge — config.toml + deploy (rule 96 #11).

### 5.3 Frontend (CC-B)
- Баннер на `/tutor/schedule` + Radix sheet. `LessonDetailsDialog` 3-кнопочный flow и `handleCompleteLesson` **не трогать** (coexist, rule 10). Reuse `calculateLessonPaymentAmount`, `getLessonParticipants`, `cancelLesson`, `LessonMaterialsDrawer`, `formatCurrency` (копейки `/100`).

## 6. UX
- Один primary CTA «Подтвердить все»; «Позже» (dismiss до следующего захода). Lucide (без эмодзи), socrat-токены (rule 90). Суммовые inputs 16px + `touch-action:manipulation` (rule 80). `React.memo` строки, lazy sheet, без framer-motion (performance.md).

## 7. Validation (AC)
- **AC-1:** при заходе с прошедшими `regular booked` (>3ч, <14д) — баннер с верным N; иначе нет.
- **AC-2:** индивид → «Подтвердить» = `completed` + 1 pending `tutor_payments` на (дефолт/edited) сумму; «не состоялось» → `cancelled`, 0 платежей.
- **AC-3:** группа → «был» сумма>0 → payment; «не был»→0 → НЕТ payment; edited суммы применяются; атомарно.
- **AC-4:** до «Подтвердить» — НИ ОДНОЙ строки `tutor_payments` / смены статуса (проверка БД).
- **AC-5:** отмена подтверждённого → payment сторнирован.
- **AC-6:** не трогает trial/mock_exam/consultation и занятия вне окна (3ч/14д).

## 8. Cross-cutting invariants
- **#3 (главный):** деньги/outbound — только на «Подтвердить». До этого ноль записей.
- **rule 10:** `TutorSchedule.tsx` high-risk — только баннер + entry + state; complete/cancel/create не менять.
- **rule 60 (оплаты):** идемпотентность `(lesson_id, tutor_student_id)`; статусы pending/paid; дата = `start_at`; без overdue.
- **rule 80 / 90 / performance / 97** (см. §6, §5.2).

## 9. Risks
| Риск | Митигация |
|---|---|
| Переплата за no-show | «не был» → дефолт 0; «не состоялось» → 0 платежей; гейт подтверждения |
| Регрессия платёжной RPC | reuse, не менять текущий 3-кнопочный flow; Codex + AC-2..5 |
| Регрессия расписания (TutorSchedule high-risk) | rule 10 минимум; smoke + ручная QA |
| «Дефолт для всех» создаёт лишние оплаты | оплата только на явное «Подтвердить» (не молча) |
