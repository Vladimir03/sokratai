## Section 0: Job Context

**Core Jobs (AJTBD Job Graph, tutor side):**

- **R1 — Проверка ДЗ и оценка прогресса ученика** (primary)
  - R1-2: понять, где ученик ошибается
  - R1-4: видеть прогресс во времени (частично — см. Non-Goals)
- **R2 — Обратная связь ученику** (secondary)
  - Быстрая похвала / напоминание через Telegram
- **R3 — Планирование следующего урока** (secondary)
  - Паттерны по задачам как вход в корректировку плана
- **R4-1 — Быстро собирать качественное ДЗ** (indirect)
  - Итоги по задаче как сигнал качества задачи из KB

**Триггер:** репетитор открывает ДЗ после того, как часть группы сдала (утро понедельника перед уроком, вечер перед звонком родителю, перед планированием следующего занятия).

**Жалобы репетиторов (цитаты из тикета):**

1. «А где я могу посмотреть, что правильно, что неправильно в ДЗ и сколько баллов (сколько использовано подсказок)» — вопрос #2 от второго репетитора
2. «Статистика нужна в формате ЕГЭ по задачам, по каждому ученику отдельно, важна динамика ученика vs самого себя» — Женя, звонок 2026-04-01

**Wedge alignment:** прямое усиление wedge «AI = draft + action». Статистика без действия = слоп; статистика с one-click CTA (Telegram-напоминание, похвала, переход к переписке, правка балла) = реальная экономия 10–15 минут на review группы.

**Пилотный фильтр:** GO, P1 в батч-тикете. Не блокирует пилот, но сильно усиливает value prop для удержания репетиторов.

## Problem Statement

Репетитор, выдающий ДЗ группе из 5–10 учеников, **не имеет одного экрана**, на котором видно: кто сдал, кто не сдал, кто сколько баллов набрал по каждой задаче, сколько использовал подсказок, и где именно ошибался. Чтобы сейчас это выяснить, репетитор открывает переписку каждого ученика поочерёдно — это 10–15 минут на группу, и инсайты уровня «эту задачу провалили все» теряются, потому что их невозможно увидеть поперёк учеников. В результате: (а) репетитор не успевает подготовиться к уроку, (б) корректировка плана происходит по наитию, а не по данным, (в) часть учеников теряется в воронке (не приступал → не напомнили → забили).

---

## Goals

1. **Закрыть R1-2 за 30 секунд:** репетитор открывает ДЗ и за полминуты понимает, что произошло в группе и с каждым учеником — без открытия отдельных переписок.
2. **Сделать действие частью статистики:** каждая выявленная проблема сопровождается one-click CTA (напомнить в Telegram, открыть переписку с фильтром по задаче, изменить балл, похвалить).
3. **Дать репетитору честные частичные баллы:** балл задачи в формате ЕГЭ (0..max_score), не бинарный verdict, с возможностью ручной корректировки.
4. **Сохранить вертикальные паттерны по задачам:** репетитор видит, какие задачи провалила вся группа — это вход в R3 и R4-1.
5. **Не ломать существующий `GuidedThreadViewer`:** drill-down использует уже работающий компонент с фильтром по задаче (E8/E9), без параллельной реализации.

---

## Non-Goals

1. **Сравнение учеников между собой / лидерборд.** Женя явно сказал «не нужно». Не строим рейтинги, не показываем чужие баллы одному ученику. Лишняя социальная динамика обесценивает продукт для целевого сегмента.
2. **Прогресс по номерам КИМ во времени (Job B).** Динамика «ученик vs сам себя» по нескольким ДЗ — это отдельная фича внутри карточки ученика, не на странице результатов ДЗ. Вынесено в P2 / отдельную спеку.
3. **Аналитический дашборд `/tutor/analytics`.** Большая страница с фильтрами по периоду, мульти-ДЗ графиками. Явно KILL для пилота: не усиливает wedge, не просился, дорого.
4. **AI-генерация «разбора урока» или «рекомендаций по плану».** Никаких AI-эссе в блоке «Требует внимания» — только детерминированные правила. AI в этом UI участвует только как источник частичного балла и короткого комментария к задаче.
5. **Экспорт в Excel/PDF.** Данных о спросе нет. Если попросят — добавим отдельным тикетом.
6. **Realtime-обновление хитмапа.** Refetch при открытии и invalidation при ручной правке балла. Realtime ради статистики — не стоит бюджета.
7. **Полная мобильная оптимизация хитмапа.** Базовая адаптация (горизонтальный скролл, sticky-имя) — да. Специальная collapsed-mobile view — в P1.

## Target Users

**Primary persona: репетитор физики ЕГЭ/ОГЭ** (Женя, второй пилотный репетитор, аналогичный сегмент). Работает индивидуально с 5–15 учениками, платит сам за инструмент или платит организация, ставки 3000–4000₽/час. Знает neofamily / РЕШУ ЕГЭ как бенчмарки. Открывает Сократ с десктопа утром и с телефона между занятиями.

**Secondary persona:** репетитор по математике (тот же wedge). Формат ЕГЭ и частичные баллы релевантны так же.

**Не в целевой аудитории:** школьные учителя с классом 30+, репетиторы начальной школы, преподаватели групповых курсов-потоков.

## User Stories

### Primary — быстрый review группы

1. **Как репетитор**, я хочу **за 10 секунд увидеть, кто из группы сдал, а кто нет**, чтобы знать, кого пинговать до урока. *(R1-2, R2)*
2. **Как репетитор**, я хочу **одним взглядом увидеть среднюю успеваемость группы** (сдали 4/5, средний балл 12/16), чтобы понять, было ли ДЗ по силам. *(R1-2, R3)*
3. **Как репетитор**, я хочу **увидеть матрицу учеников × задач с баллами**, чтобы найти паттерны: и горизонтальные (ученик в беде), и вертикальные (задача провалена всей группой). *(R1-2, R3, R4-1)*
4. **Как репетитор**, я хочу **отличать «решил без подсказок» от «решил с тремя подсказками»**, даже если балл одинаковый, чтобы правильно оценивать готовность ученика к ЕГЭ. *(R1-4)*

### Primary — action layer

5. **Как репетитор**, я хочу **одной кнопкой напомнить в Telegram** ученику, который не приступил к ДЗ, не копируя ссылку вручную. *(R2)*
6. **Как репетитор**, я хочу **одной кнопкой похвалить в Telegram** ученика, который решил всё без подсказок. *(R2, удержание)*
7. **Как репетитор**, я хочу **видеть список «требует внимания» с детерминированными сигналами** (не приступал, задача провалена группой, ученик с большим числом подсказок), чтобы не пропустить важное. *(R1-2, R3)*

### Primary — drill-down

8. **Как репетитор**, я хочу **раскрыть ученика inline** (не переход на новую страницу) и увидеть полоску всех его задач с баллами, чтобы быстро переключаться между учениками. *(R1-2)*
9. **Как репетитор**, я хочу **кликнуть на конкретную задачу и увидеть переписку guided-чата**, отфильтрованную по этой задаче, чтобы понять, где ученик споткнулся. *(R1-2)*
10. **Как репетитор**, я хочу **вручную исправить балл**, если я не согласен с оценкой AI, потому что я профессионально отвечаю за итоговую оценку. *(R1-2, доверие к продукту)*
11. **Как репетитор**, я хочу **видеть, что AI-оценка сохраняется после моей правки** (как «draft» рядом с «final»), чтобы потом иметь возможность сравнить и откалибровать AI. *(прозрачность, не теряем данные)*

### Edge cases

12. **Как репетитор**, я хочу **видеть ДЗ, которое никто ещё не сдал**, с понятным пустым состоянием, а не ошибкой или пустым экраном.
13. **Как репетитор**, я хочу **открыть страницу с телефона** и хотя бы прочитать шапку-сводку и список «требует внимания», даже если хитмап приходится скроллить.
14. **Как репетитор**, я хочу **видеть ДЗ, у которого часть учеников сдала, а часть ещё нет**, без предположения, что все уже дошли до конца.

## Acceptance Criteria (testable)

Каждый AC ниже = действие, которое возвращает PASS/FAIL при ручном или автоматическом тесте.

- **AC-1:** Открыв `/tutor/homework/:id` (canonical) или `/tutor/homework/:id/results` (legacy → redirect) для ДЗ, где ≥1 ученик сдал, репетитор видит шапку-сводку (сдали X/Y, средний балл, не приступали, требует внимания) **в течение одного scroll-screen** без открытия отдельных переписок. **Semantic invariant:** метрика «Требует внимания» в шапке = `notStarted + per_student.filter(s => s.needs_attention).length`. Backend считает `needs_attention` только для сдавших (`lowScore < 30%` OR `hint_overuse >= ceil(tasks*0.6)`), поэтому фронтенд явно добавляет `notStarted` — иначе шапка не согласуется с action block ниже.
- **AC-2:** В таблице хитмапа `score === null` рендерится серой клеткой с прочерком; `score / max ≥ 0.8` — зелёная; `0.3..0.8` — жёлтая; `< 0.3` — красная. Проверяется на seed-данных с 4 разными ситуациями в одной строке.
- **AC-3:** Клик по строке ученика раскрывает inline drill-down с полоской мини-карточек задач и `GuidedThreadViewer` с `taskFilter = selectedTaskId`. Только один ученик раскрыт одновременно.
- **AC-4:** Клик по мини-карточке задачи в drill-down меняет `selectedTaskId`, `GuidedThreadViewer` ремоунтится (`key={taskId}`), скролл сбрасывается наверх треда.
- **AC-5:** Открытие модалки правки балла → ввод значения `0..max_score` (шаг 0.5) → «Сохранить» → в БД появляются `tutor_score_override`, `tutor_score_override_at`, `tutor_score_override_by`; `ai_score` не затирается; React Query инвалидируется на 3 ключа; в хитмапе клетка обновляется.
- **AC-6:** Для ученика без `submitted` в шапке отображается chip «N не приступали»; в блоке «Требует внимания» появляется danger-пункт с CTA «Напомнить в Telegram»; клик по CTA открывает Dialog с preset-сообщением и редактируемой textarea.
- **AC-7:** Для ученика без Telegram-связки CTA «Напомнить» подменяется на «Напомнить на email» (тот же Dialog-pattern, использует существующий email queue). Если у ученика **ни Telegram, ни email**, кнопка дизейблится с tooltip «Нет каналов для уведомления». Внутри `RemindStudentDialog` рендерятся tabs `[Telegram] [Email]`; недоступные каналы — disabled. Backend `POST /assignments/:id/students/:sid/remind` принимает optional `channel: 'auto' | 'telegram' | 'email'` — explicit значение отключает cascade-fallback.
- **AC-8:** Backfill-миграция применяется без downtime: для существующих `homework_tutor_task_states` `ai_score = (verdict === 'CORRECT' ? max_score : 0)`. Существующие ДЗ открываются без ошибок.
- **AC-9:** Для нового guided chat на `detailed_solution` задаче `evaluateStudentAnswer` возвращает `{ verdict, ai_score, ai_score_comment }`, где `ai_score ∈ [0, max_score]` с шагом 0.5; `ai_score` сохраняется в `homework_tutor_task_states` в реальном времени.
- **AC-10:** Telemetry-события `results_v2_opened`, `drill_down_expanded`, `manual_score_override_saved`, `telegram_reminder_sent_from_results` уезжают в `window.dataLayer` с payload без PII (только id и числа). Проверяется через DevTools при ручном прогоне.
- **AC-11:** `npm run lint && npm run build && npm run smoke-check` проходит зелёным после всех изменений на Windows + Chrome. На Safari 15+ и iOS Safari страница рендерится без layout breaks (горизонтальный скролл хитмапа работает, sticky-имя ученика работает).

## Requirements

### Must-Have (P0)

#### P0-1. Шапка-сводка ДЗ

Однострочная (переносится на мобиле) сводка с метриками:

- Сдали X из Y
- Средний балл (сумма / max_total, в формате `12.4 / 16`, `70%`)
- Среднее время в guided-чате (минуты)
- Среднее число подсказок на ученика
- Отдельный красный chip «N не приступал/и» (если есть)

**Acceptance:**
- [ ] Отображается над всеми остальными блоками
- [ ] Метрики считаются только по ученикам, у которых есть хотя бы одна попытка (submitted = true)
- [ ] Процент округляется до целого
- [ ] При 0 сдавших показывает пустое состояние «Никто ещё не сдал ДЗ»
- [ ] Chip «не приступали» кликается и ведёт к фильтрации хитмапа (nice) или скроллит к блоку «Требует внимания»

#### P0-2. Блок «Требует внимания» (action block)

Детерминированный список из 0–N пунктов, сгенерированный по правилам:

- **Danger:** ученик не приступал → CTA «Напомнить в Telegram»
- **Warning:** задача провалена (≥3 сдавших, solvedCount / submitted ≤ 25%) → CTA «Разобрать на следующем занятии» (no-op в P0, только визуально)
- **Warning:** ученик набрал ≥75%, но потратил ≥`HINT_OVERUSE_THRESHOLD` подсказок → CTA «Открыть переписку». Порог: `HINT_OVERUSE_THRESHOLD = Math.ceil(tasks.length * 0.6)` (для 8 задач — 5, для 26 — 16).
- **Success:** ученик набрал max_total без подсказок → CTA «Похвалить в Telegram»

**Acceptance:**
- [ ] Правила реализованы детерминированно на фронтенде (не AI)
- [ ] Пороги вынесены в константы (легко калибровать после пилота)
- [ ] Кнопки «Напомнить / Похвалить в Telegram» открывают `Dialog` с **preset-сообщением + редактируемой textarea**, отправка по подтверждению. Никакой fire-and-forget.
- [ ] Если у ученика нет Telegram-связи — CTA подменяется на «Напомнить на email» (preset+edit, через существующий email queue), а не скрывается
- [ ] Блок не показывается, если список пуст (success case)

#### P0-3. Хитмап учеников × задач

Таблица: строки = ученики, столбцы = задачи. Sticky-колонка имени слева, горизонтальный скролл для задач.

**Клетка:**
- Содержимое: `score/max` (цифра, всегда, включая `1/1`)
- Фон и бордер по доле `score/max`: зелёный ≥80%, жёлтый 30–79%, красный <30%, серый при `score === null` (не приступал)
- Иконка подсказки (Lucide `Lightbulb`) в правом верхнем углу если `hints ≥ 1`, с числом рядом если `hints ≥ 3`
- Title (hover tooltip): `Балл: X/Y · подсказок: Z`

**Заголовок столбца:** `№{order}` + `КИМ {kim_number}`
**Правая часть строки:** `{score}/{max_total}` · подсказки (суммарно) · время (мин)
**Итог по столбцу (footer row):** `avg.toFixed(1)/max` + `solvedCount/submittedCount`, красная подсветка если `avg/max < 0.4` и `submitted ≥ 3`

**Acceptance:**
- [ ] Клик по строке ученика раскрывает drill-down inline (см. P0-4)
- [ ] Только один ученик раскрыт одновременно
- [ ] Sticky-имя работает при горизонтальном скролле
- [ ] На мобиле (< 768px) работает горизонтальный скролл, имя остаётся sticky
- [ ] Пустая клетка (ученик не дошёл до задачи) = серая с прочерком `—`
- [ ] LaTeX в заголовках задач: в хитмапе используем `stripLatex` (plain-text), в drill-down — полный `MathText`

#### P0-4. Drill-down через `GuidedThreadViewer`

При раскрытии ученика показывается:

1. **Полоска мини-карточек задач** (горизонтально, скроллится):
   - Для каждой задачи: `№{order} · КИМ {kim}`, статус-иконка, балл в цветной плашке, подсказки, попытки, маленькая иконка карандаша
   - Клик по карточке → выбирает задачу
   - По умолчанию выбирается первая проблемная задача (балл < 80% от max), fallback — первая задача
   - Выбранная карточка обведена `ring-2 ring-slate-800` (нейтральный, чтобы не создавать «ложно зелёное» поверх красной клетки)

2. **`GuidedThreadViewer`** с `taskFilter = selectedTaskId` и `studentId = expandedStudentId`:
   - Переиспользуется существующий компонент (E8: блок «Условие задачи», E9: realtime INSERT subscriptions)
   - Ремоунт при смене `selectedTaskId` через `key={taskId}` для закрытия висящих диалогов
   - Над viewer-ом — compact header: задача, балл, подсказки, попытки, короткий AI-комментарий (если есть), кнопка «Изменить балл»

3. **Для не приступавших ученикови:** вместо полоски и viewer-а — одна карточка с текстом «Не приступал к ДЗ» и CTA «Напомнить в Telegram»

**Acceptance:**
- [ ] `GuidedThreadViewer` получает `taskFilter` и фильтрует сообщения по выбранной задаче
- [ ] Realtime subscription (E9) продолжает работать в drill-down
- [ ] Блок «Условие задачи» внутри viewer-а показывается как сейчас (collapsible, click-to-zoom)
- [ ] При смене `selectedTaskId` viewer ремоунтится и скролл сбрасывается наверх треда
- [ ] При смене раскрытого ученика предыдущий drill-down закрывается полностью
- [ ] Mini-карточки задач скроллятся горизонтально на мобиле

#### P0-5. Ручная правка балла (модалка)

`Dialog` (shadcn/ui), открывается из двух мест:
- Иконка карандаша на мини-карточке задачи (быстро)
- Кнопка «Изменить балл» в header drill-down viewer-а (informed)

**Содержимое модалки:**
- Заголовок: `Задача №{order} · КИМ {kim}`
- Текущая AI-оценка: `AI: {score}/{max}` (read-only, как draft)
- Input (`type="number"`, `min=0`, `max={max_score}`, `step=0.5` для поддержки половин баллов) для нового балла
- Textarea (optional) — tutor_comment
- Кнопки: `Отмена` / `Сохранить`

**После сохранения:**
- В БД сохраняется `tutor_score_override`, `tutor_score_override_comment`, `tutor_score_override_at`, `tutor_score_override_by`
- AI-оценка (`ai_score`, `ai_comment`) **не затирается** — сохраняется рядом
- В UI везде, где показывается балл задачи (хитмап, мини-карточка, header viewer-а, сводка группы), используется `final_score = tutor_score_override ?? ai_score`
- Рядом с баллом, если есть override, показывается маленький badge «правка репетитора» (tooltip: `AI: X/Y`, `Репетитор: A/B`, `Комментарий: ...`)
- Ученик на своей стороне видит `final_score` и, если есть, `tutor_score_override_comment`

**Acceptance:**

## Success Metrics

### Leading indicators (1–4 недели после запуска)

| Метрика | Целевой порог | Как мерить |
|---|---|---|
| **% ДЗ, где репетитор открывает Results v2** | ≥80% от ДЗ, где хотя бы 1 ученик сдал | Событие `results_v2_opened` / число sent assignments с ≥1 submission |
| **Среднее время на странице Results** | ≥30 сек (есть вовлечённость) и ≤5 мин (нет залипания) | Heartbeat telemetry |
| **% ДЗ, где использован drill-down** | ≥50% открытых Results | Событие `drill_down_expanded` |
| **% ДЗ, где сработал CTA «Напомнить в Telegram»** | ≥30% от ДЗ с не приступавшими | Событие `telegram_reminder_sent_from_results` |
| **% заданий, где была ручная правка балла** | 5–30% (широкий диапазон) | Событие `manual_score_override_saved` / число заданий |

**Почему диапазон для правки балла:** слишком низкий процент (<5%) = репетитор либо не знает, что правка есть, либо слепо доверяет AI. Слишком высокий (>30%) = AI-оценка недоверчива, надо чинить evaluator. Оба экстрема — сигналы.

### Lagging indicators (1–3 месяца)

| Метрика | Целевой порог | Как мерить |
|---|---|---|
| **Retention репетиторов-пилотов** | +10pp vs baseline без Results v2 | Cohort retention по недельной активности |
| **Заявленная экономия времени на review ДЗ** | ≥10 минут на группу (qual) | Интервью с Женей + вторым репетитором через 2 недели после запуска |
| **NPS «насколько полезен экран результатов ДЗ»** | ≥7/10 | Встроенный вопрос / интервью |
| **Количество ДЗ, выданных репетитором в неделю** | +20% | Стат по `homework_tutor_assignments` |

**Качественная метрика-сигнал (самая важная):** когда спросим Женю «если бы завтра выключили Results v2, как бы ты работал?», ответ должен быть «больно» или «не готов». Если ответ «ну, нормально, как раньше» — фича провалилась, даже если метрики сошлись.

## Open Questions — Resolved (2026-04-07)

1. **[Engineering] Хранение балла → Вариант A.** Две отдельные колонки `ai_score` / `tutor_score_override`, `final_score = COALESCE(override, ai_score)`. Сохраняет историю AI-оценки для будущих evals («согласие AI vs репетитор»). См. P0-5, P0-6. ✅

2. **[Product+Data] Методология балла → стандарт ЕГЭ, без своей шкалы.**
   - **Источник истины:** `homework_tutor_tasks.max_score` (уже есть в БД, задаётся в конструкторе ДЗ). Для задач из KB — это первичные баллы ФИПИ; для свободных — репетитор сам ставит max при добавлении задачи.
   - **Для `short_answer`:** AI возвращает 0 или `max_score` (бинарно, как сейчас). Никакой собственной методики.
   - **Для `detailed_solution`:** AI возвращает целое или `0.5`-step значение `0..max_score` с коротким комментарием «почему не максимум». Промпт явно отсылается к ФИПИ-критериям, а не к нашей внутренней шкале.
   - **Olympiad сегмент (Егор):** работает на той же модели — `max_score` per-task конфигурируется в конструкторе. Для олимпиад репетитор может выставить `max_score = 10` (или сколько нужно) и опционально перейти на pass/fail (`max_score = 1`). Никакого отдельного режима.
   - **Hints + время в guided-чате** — отображаются всегда как proxy-метрики готовности, но **не входят в балл**. Это снимает риск, что ученик получит «-2 балла за подсказки» и это будет несовместимо с ЕГЭ-семантикой.
   - **Почему так:** одна модель работает для обоих сегментов, формат знаком репетитору ЕГЭ/ОГЭ (= ноль обучения), не требует «защищать» внутреннюю шкалу перед олимпиадниками. ✅

3. **[Product] Похвалить/Напомнить → preset+edit.** Dialog с заполненным сообщением и редактируемой textarea, отправка по кнопке. Никакой fire-and-forget. См. P0-2 acceptance. ✅

4. **[Product] Порог «много подсказок» → `ceil(tasks.length * 0.6)`.** Константа `HINT_OVERUSE_THRESHOLD` в коде, легко калибровать после пилота. См. P0-2. ✅

5. **[Design] Ring выбранной мини-карточки → `ring-2 ring-slate-800`.** Нейтральный, чтобы не давать «ложно зелёное» поверх красной клетки. См. P0-4. ✅

6. **[Design] 26 задач на десктопе.** Оставляем как на мокапе: фиксированная высота клетки 44px, минимальная ширина 56px, горизонтальный скролл от ~720px ширины таблицы. Smoke-test на мокапе 26 × 10 — добавить проверочную точку перед демо Жене. ✅ (с проверкой)

7. **[Engineering] Invalidate vs merge при правке балла → `invalidateQueries`.** Manual edit — редкое (pилотно ≤30 раз/неделю), не realtime, latency терпит refetch. Merge оправдан только для high-frequency обновлений (как realtime thread INSERT). Простота и корректность важнее. См. P0-5 acceptance — три ключа. ✅

8. **[Product] Нет Telegram → email-версия CTA.** Кнопка превращается в «Напомнить на email» (preset+edit, через существующий email queue из `_shared/email-sender.ts`). Action должен быть всегда. См. P0-2. ✅

9. **[Data] Telemetry → используем `src/lib/homeworkTelemetry.ts`.** Инфраструктура уже есть: gtag/dataLayer (`window.dataLayer`, `window.gtag`) с типизированным `GuidedTelemetryEvent`. Подключение бесплатное → telemetry промоут из P1 в **P0-7**. Новые события: `results_v2_opened`, `drill_down_expanded`, `manual_score_override_saved`, `telegram_reminder_sent_from_results`. Без PII в payload. ✅
