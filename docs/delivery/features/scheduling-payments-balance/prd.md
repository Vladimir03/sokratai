## 1. Job Context

- **Segment**: B2B, репетиторы физики ЕГЭ/ОГЭ (платный пилот), ведут 10+ учеников и/или мини-группы. **Платит родитель (ЛПР) офлайн** (нал/перевод/СБП); **ученик** — пользователь; **репетитор (ЛВР)** ведёт учёт. Сократ **НЕ проводит платежи** — это личная бухгалтерия репетитора.
- **Core Job**: R4 — «сохранять контроль и не утопать в рутине при масштабировании» (admin/контроль-ветка Core Job, doc 14 эмоциональный слой: *не выгорать, не чувствовать себя заваленным рутиной, ощущать контроль*).
- **Sub-jobs (ветка «деньги/админ»)**: запланировать занятия наперёд · зафиксировать проведение · назначить цену без дробей · **знать, кто сколько должен** · зафиксировать полученные деньги · **не терять деньги/долги по забывчивости** · **спокойно говорить с родителем о деньгах** · продавать абонементы.
- **Wedge alignment**: **НЕ wedge.** Wedge = «ДЗ по теме за 5-10 мин» (memo 08). Это **retention / anti-churn supporting job**. Размер инвестиции = убрать боль, из-за которой репетитор бросит инструмент или перестанет доверять, **не** строить бухгалтерию/CRM. Делаем **узкий срез**: личный учёт репетитора + read-only выписка для родителя — НЕ CRM, НЕ автосообщения, НЕ родительский логин (канон: «CRM / сообщения родителям» вне пилота).
- **Pilot impact**: если репетитор теряет деньги/нервы на ручном учёте — он не продлит пилот. Деньги-боль бьёт в эмоциональный слой Core Job напрямую.

## 2. Problem

Сейчас в Сократе учёт = «оплачено/не оплачено» по каждому занятию (`tutor_payments`, статусы pending/paid). Реальные боли (цитаты Егора):

- **Ручной подсчёт за период.** «Недели не целые… каждый раз репетитору надо вручную посчитать, сколько было занятий и сколько суммарно должен заплатить родитель за период… это много возни». Родитель платит 1–2 раза в месяц, а занятия идут по сетке.
- **Деньги не привязаны к занятию в голове репетитора.** «Родитель внёс 10 000 — я записал, что получил 10 000. За какое конкретно занятие пошли деньги — мне не хочется разбираться. Хочу, чтобы кто-то посчитал за меня».
- **Нет единого числа долга.** Нельзя за секунду ответить «сколько должен ученик X»; нет места для уверенного разговора с родителем.
- **Дроби и устаревшие суммы.** Цена = ставка×длительность → дроби на 1,5 ч; у групп `payment_amount` — снимок, «протухает» при правке (поэтому правка длительности групп заблокирована, rule 60).
- **Посещаемость смешана с оплатой.** «С некоторыми родителями договор: пропущенные — 50%, с кем-то — 100% (ученик посмотрит видео + сделает ДЗ)… репетитор должен иметь возможность вписать стоимость за конкретное занятие в окошко, и это не должно быть связано с тем, посетил ученик или нет». Учёт посещаемости = дисциплина; учёт оплат = деньги. Это **разные** вещи.
- **`/pay`-бот мыслит занятиями**, группы недо-напоминаются.

Workaround сегодня: репетитор считает в тетрадке/Excel/голове → ошибки, забытые долги, неловкие разговоры с родителем.

## 3. Solution

**Модель кошелька (ledger).** У каждого ученика — **баланс ₽ = Σ пополнения − Σ списания**. Зелёный «+» (оплачено вперёд) / красный «−» (должник).

- **Списания** капают по занятиям (debit-запись на занятие), сумма = **редактируемое ₽-поле**, дефолт = цена ученика. **Не привязаны к посещаемости** — репетитор вписывает что договорились (100/50/0/custom).
- **Пополнения** — крупная запись «получил ₽» (от родителя), **без раскидки по занятиям**.
- Система **сама держит итог**. Репетитор смотрит **одно число** и **одну выписку**.

**Поверхности (см. макет):**
- **Карточка ученика → вкладка «Обзор»** (бывш. «Прогресс», переименована): сверху **компактная карточка-сводка** (Баланс ± · «Внести оплату» · «Отчёт родителю» · «Все операции →»), ниже — учёба (проверка/цель/работы). Деньги видны «за один взгляд» вместе с прогрессом.
- **Вкладка «Оплаты»**: полный реестр (лента операций, ближайшие списания, редактируемые списания).
- **«Внести оплату»**: одно поле ₽ + дата, из карточки **и** из списка «должников».
- **Отчёт родителю**: **ОДИН общий** read-only отчёт (сверху прогресс кратко: цель + балл + динамика + последние работы; снизу баланс + выписка по датам + ИТОГ), по **share-ссылке** (реюз паттерна `/p/:slug`). Родитель не логинится. Без решений/критериев (anti-leak).
- **Должники = отрицательный баланс** (не «неоплаченные занятия»); бот: «у X баланс −1 600 ₽».
- **Списание появляется авто** при наступлении занятия (= цена) + **ежедневная сводка** «начислено за день» с правкой/undo. Будущие занятия не списываются.

**Зафиксированные решения (locked, со-дизайн через AskUserQuestion):**
1. Списание-дефолт = **правило на ученика** (дефолтная цена ученика). Раз посещаемость отложена — пропуски/скидки правятся **вручную в окошке**; авто-% за пропуск вернётся вместе с посещаемостью.
2. **Посещаемость — НЕ в этом раунде** (отдельный раунд позже).
3. Пополнение = **одно поле ₽** из карточки + из «должников», без раскидки.
4. Отчёт родителю — **один общий** (прогресс + баланс), share-ссылка; прогресс-часть **кратко** (цель + балл + динамика + последние работы), без решений/критериев.
5. Деньги во вкладке «Обзор» — **карточка-сводка сверху**, полный реестр на «Оплаты» (без дублирования ленты).
6. Списание авто при наступлении + ежедневная сводка с undo; абонемент в дополнение к поурочным; полная замена paid/unpaid → баланс; фундамент «занятие+участники + цена-на-ученике».

## 4. Scope

### IN (делаем — это v1 = «Phase 2: баланс»)
- **Ledger**: append-only `tutor_ledger_entries` (списание/пополнение/корректировка) + денормализованный `tutor_students.balance_cents` (копейки).
- **Цена-на-ученике** (`price_cents`, копейки) — фундамент (Phase 1, частично уже в parity-раунде).
- **Списание** = editable amount per занятие, дефолт = цена ученика, **decoupled от посещаемости**.
- **Пополнение** = одно поле ₽ из карточки ученика + из «должников».
- **«Обзор»** карточка-сводка баланса + **«Оплаты»** полный реестр.
- **Авто-списание** при наступлении занятия + **ежедневная сводка** (правка/undo); будущие не списываются.
- **Reframe** `/pay`-бота и «Подтверждение прошедших» → баланс/пополнения.
- **Один общий отчёт родителю** (прогресс lite + баланс) read-only share-ссылка.
- **Seed-миграция** `tutor_payments` → стартовые балансы (reconciliation-гейт).

### OUT (НЕ делаем — осознанный отказ)
- **Платёжный процессинг** (карты/СБП/эквайринг) — мы не финтех; деньги ходят офлайн.
- **Полный CRM / автосообщения родителям / родительский логин** (канон: вне пилота).
- **Трекинг посещаемости** (был/пропустил/перенёс) — отдельный раунд.
- **Аналитика/дашборды** по деньгам.

### LATER (потом)
- **Абонемент** (Phase 3): объект расписания, фикс сумма в фикс день месяца, участники, авто-списание, флаг «покрыто абонементом». Брать только если пилотный репетитор попросит.
- **Посещаемость** + авто-% за пропуск (привязка дефолта списания к «пропустил»).
- Multi-subject / school-target в отчёте.

## 5. User Stories

**Репетитор (primary):**
- Когда я открываю ученика, я хочу за 2 секунды увидеть и долг, и как идёт учёба — чтобы решить, писать ли родителю.
- Когда занятие прошло, я хочу, чтобы списание появилось само (= цена) — чтобы не вести учёт вручную.
- Когда был пропуск/скидка, я хочу вписать сумму в окошко — независимо от того, ходил ученик или нет.
- Когда родитель передал деньги, я хочу записать одно число — не разбираясь, за какие занятия.
- Когда родитель спрашивает «сколько мы должны», я хочу показать/отправить выписку за 5 секунд — уверенно, по факту.
- Когда у ученика отрицательный баланс, я хочу видеть его в «должниках» и напомнить (бот) — чтобы не работать бесплатно по забывчивости.

**Родитель (read-only consumer):**
- Когда репетитор прислал ссылку, я хочу увидеть прогресс ребёнка + сколько должен — без логина, понятно.

**Ученик**: не затронут (деньги — репетитор↔родитель).

## 6. Success Criteria

**Leading (3–7 дней после выката):**
- ≥1 пополнение записано репетитором на 2-й день использования.
- Репетитор открыл баланс ученика (вкладка «Обзор») ≥1 раз.
- Время записать полученную оплату < 5 сек (1–2 тапа) — наблюдение/таймер.
- Отправлена ≥1 выписка родителю (share-ссылка открыта).

**Lagging (2–4 недели):**
- Репетитор **перестал вести параллельную тетрадку/Excel** по деньгам (сигнал переключения — Замесин: уход от текущего решения).
- Баланс в app **сходится** с реальной тетрадкой репетитора (reconciliation на созвоне).
- Егор/Эмилия говорят «теперь не считаю период вручную» (qualitative).
- Снизилось число «потерянных» долгов (ученик ушёл должным).

**Anti-metric:** не создали БОЛЬШЕ рутины — если усилие на внесение пополнений > боль от подсчёта, модель провалилась.

## 8. Risks

- **Money-correctness (highest).** Неверные единицы (копейки/рубли), двойное списание, ошибка seed → искажённые балансы → потеря доверия. Митигация: партиал-unique-on-active идемпотентность + `ON CONFLICT`, единый `*100` в seed, reconciliation-гейт до cutover, прогон cron дважды = баланс не меняется.
- **Доверие к авто-списанию.** «Само списалось» может пугать. Митигация: ежедневная сводка + undo в 1 тап; формулировка «бухгалтерская запись, не платёж».
- **Не вносят пополнения** (RAT-2) → баланс врёт за неделю. Митигация: внесение в 1 тап из «должников»; мерить % с пополнением.
- **Scope creep в CRM/бухгалтерию.** Митигация: OUT-список жёсткий; родителю только read-only выписка; Принцип 15 (помогает ли пилоту?).
- **Высокорисковые файлы** (rule 10): `TutorSchedule.tsx`, `TutorPayments.tsx`, `/pay`-бот, `TutorStudentProfile.tsx`. Аддитивно, ядро не рефакторить; money-RPC через SECURITY DEFINER.
- **Backward-compat**: старые `tutor_payments` заморозить read-only (additive DB), не дропать.
