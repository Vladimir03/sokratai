# PRD: Формат проверки из KB + развернутое решение

**Tickets**: Е1 (KB → ДЗ: баллы + формат из базы) + Е4 (формат «развернутое решение»)
**Приоритет**: P1
**Дата**: 2026-04-01

---

## 1. Job Context

- **Core Job**: R4 — Сохранение контроля и качества при масштабировании
- **Sub-job**: R4-1 — Быстро собрать качественное ДЗ по теме урока
- **Вторичный Job**: R1 — Автоматическая проверка ДЗ (R1-3 классификация ошибок, R1-4 персональная обратная связь)
- **Segment**: репетиторы физики ЕГЭ/ОГЭ с 10+ учениками
- **Wedge alignment**: Да — убирает 2 ручных шага при сборке ДЗ из KB (настройка баллов + формата)
- **Pilot impact**: Егор прямо запросил. Ученики второй части ЕГЭ должны показывать ход решения — без этого AI не может классифицировать ошибки.

---

## 2. Problem

Сейчас при добавлении задачи из KB в ДЗ:

- `max_score` хардкодится в `1` — хотя в KB уже есть `primary_score` (1-4 балла по КИМ ЕГЭ). Репетитор вынужден вручную ставить баллы для каждой задачи.
- Нет формата проверки — AI одинаково проверяет КИМ №1 (краткий ответ, 1 балл) и КИМ №25 (развернутое решение, 3 балла). Ученик может написать голый ответ на задачу второй части, и AI его примет.
- Репетитор не может потребовать от ученика показать ход решения.

**Workaround**: Егор вручную пишет в условие задачи «покажи ход решения». Это не enforcement — AI не отклоняет ответ без решения.

**Цитата**: Егор: «Для второй части ЕГЭ — формат проверки "развернутое решение", чтобы ученики показывали ход решения. Без решения бот не принимает задачу. В банке: КИМы 1-20 краткий ответ, 21-26 развернутое решение»

---

## 3. Solution

Добавить поле `check_format` в KB и в конструктор ДЗ. При добавлении задачи из KB — автоматически подгружать `max_score` (из `primary_score`) и `check_format` (из `answer_format`). Репетитор может поменять на финальном этапе.

В guided chat: если `check_format === 'detailed_solution'` и ученик прислал только короткий ответ без шагов — AI отклоняет и просит показать решение.

---

## 4. Scope

### IN (делаем)

- Новое поле `check_format: 'short_answer' | 'detailed_solution'` в `homework_tutor_tasks`
- Новое поле `check_format` в `kb_tasks` (заполняется при seed/модерации на основе `kim_number`)
- Конвертер `kbTaskToDraftTask`: прокидывать `primary_score → max_score` и `answer_format → check_format`
- UI в конструкторе ДЗ: отображать и позволять менять `check_format` и `max_score` на карточке задачи
- AI enforcement в guided chat: если `check_format === 'detailed_solution'` и ответ < порога — AI просит показать решение
- Default mapping по `kim_number`: 1-20 → `short_answer` (1 балл), 21-26 → `detailed_solution` (2-4 балла)

### OUT (не делаем)

- Автоматический парсинг задач с neofamily/РЕШУ ЕГЭ (это Ж1, отдельная фича)
- Отдельный scoring rubric с критериями (ФИПИ-стиль) — слишком сложно для MVP

### LATER (потом)

- Подробные rubrics по каждому типу задачи ЕГЭ (Е6: AI промпты физики)
- Частичные баллы (0/1/2/3 вместо pass/fail) — после валидации AI качества
- Формат проверки для математики (другая структура КИМ)

## 5. User Stories

**Репетитор (Егор):**
- Добавляю задачу КИМ №25 из каталога Сократа в ДЗ → вижу, что `max_score: 3` и `check_format: развернутое решение` подгрузились автоматически → не нужно настраивать вручную
- Хочу поменять формат для конкретной задачи → меняю в карточке задачи перед отправкой ДЗ
- Создаю задачу вручную (не из KB) → выбираю формат проверки сам

**Ученик:**
- Решаю задачу второй части ЕГЭ → пишу только «42» → AI говорит «Для этой задачи нужно показать ход решения. Напиши шаги, которые привели к ответу» → дописываю решение → AI проверяет

---

## 6. Success Criteria

### Leading (3-7 дней после релиза)

- **Adoption**: Егор создаёт ≥1 ДЗ с задачами из KB, где `max_score` и `check_format` подгрузились автоматически (не правил вручную)
- **Task completion**: AI корректно отклоняет короткие ответы на задачи с `detailed_solution` в ≥80% случаев (проверить на 10+ реальных задачах КИМ 21-26)
- **Error rate**: AI не отклоняет ложно ответы, которые содержат ход решения (false rejection < 10%)

### Lagging (2-4 недели)

- **Qualitative**: Егор подтверждает, что ученики стали присылать решения, а не голые ответы, для задач второй части
- **Retention**: Егор продолжает использовать KB → ДЗ flow (а не ручной ввод) для задач с разными форматами проверки
- **Signal от Жени**: Женя (2-й пилотный репетитор) также использует check_format при создании ДЗ

## 7. Open Questions (все закрыты)

| Вопрос | Решение | Кто решил |
|---|---|---|
| Порог «слишком короткий ответ» для `detailed_solution` — по длине или по наличию шагов? | **По длине.** Начинаем с простого порога, уточняем по feedback | product |
| Как обрабатывать задачи без `kim_number` в KB — какой default `check_format`? | **`short_answer`** — наименее ограничительный вариант, не ломает текущее поведение. Репетитор может повысить до `detailed_solution` вручную | product + engineering |
| `primary_score = null` в существующих KB задачах — заполнять миграцией или оставить fallback? | **Fallback `max_score: 1` если null.** Миграция не нужна на MVP — заполнится при seed новых задач с `kim_number` | engineering |

---

## 8. Risks

| Риск | Вероятность | Mitigation |
|---|---|---|
| AI слишком строго отклоняет ответы (false rejection) | Средняя | Порог длины мягкий, AI спрашивает а не блокирует. Репетитор может сменить формат на `short_answer` |
| Ученик пишет длинный неправильный ответ и AI его принимает как «развернутый» | Средняя | AI проверяет и содержание и формат — не только длину |
| `primary_score` не заполнен в KB задачах | Высокая | Fallback: если `primary_score = null` → `max_score = 1` (текущее поведение). Заполнить через миграцию для существующих задач с `kim_number` |
| Репетитор не понимает зачем два формата | Низкая | Tooltip в конструкторе: «Краткий ответ — число/слово, Развернутое решение — ученик покажет ход решения» |
