### Wedge-связка

- **Wedge:** AI-проверка домашних заданий для репетиторов ЕГЭ/ОГЭ (физика → расширение на русский/языки/математику).
- **Pilot impact:** завышенная AI-оценка («22/22 за неидеальное сочинение») напрямую разрушает доверие репетитора к продукту — главную ценность wedge. Если AI ставит балл как строгий эксперт ФИПИ, репетитор перестаёт перепроверять каждую работу → экономит время → остаётся в платном пилоте. Это P0-фикс доверия, а не косметика.

## 1. Summary

AI-проверка завышает баллы развёрнутых работ: модель по умолчанию доброжелательна и «округляет вверх из вежливости». Критерии (`description` каждого критерия) до промпта **доходят**, не хватает **инструкции «оценивай строго, как эксперт»**. Фича добавляет единый рычаг строгости — поле `grading_discipline` в rubric-слое (`resolveSubjectRubric`), через который уже ходят и ДЗ, и пробники Часть 2. Раскатка **по предмету**, начиная с русского сочинения (филолог калибрует текст), затем физика/французский/математика.

---

## 2. Problem

### Текущее поведение
`buildCheckPrompt` (`guided_ai.ts`) инжектит role + methodology + покритериальный блок с описаниями критериев, затем «ПРАВИЛА ОЦЕНКИ» с дружелюбным тоном («похвали верный шаг»). Нет инструкции о строгости выставления балла → модель ставит максимум при частичном/спорном соответствии.

### Боль
Филолог-репетитор: AI поставил сочинению ЕГЭ **22/22**, хотя работа «далеко не идеальная». Репетитор не может доверять оценке → перепроверяет вручную → AI-проверка не экономит время → ценность wedge не реализуется.

### Текущие «нанятые» решения
Репетитор перепроверяет каждую AI-оценку вручную (то, что продукт обещал устранить).

---

## 3. Solution

### Описание
Новое опциональное поле `SubjectRubric.grading_discipline?: string | null` — клауза «будь строгим экспертом». Контент задаётся **per-subject** в rubric-builder'е; резолвер коалесцит и **занулят на numeric**. Инжектится **только** в грейдинг-промпты (`buildCheckPrompt` ДЗ + `buildMockExamPart2Prompt` Часть 2), **никогда** в подсказки/чат (tone-split).

### Ключевые решения
1. **Единый рычаг в rubric-слое** — не дублировать строгость в каждом промпте; оба грейдинг-пути уже ходят через `resolveSubjectRubric`.
2. **Tone-split** — строго только на оценке (балл/критерии/вердикт). Подсказки (`buildHintPrompt`) и чат-обсуждение (`chat/index.ts`) остаются сократическими/поддерживающими. `feedback`-разбор сохраняет уважительный тон (строка клаузы «строгость в баллах, не в словах»).
3. **Per-subject rollout** — итерация 1 только русское сочинение № 27; физика/языки/математика → `null` (no-op), включаются после валидации.
4. **Generic клауза + условный грамотность carve-out** (review fix #2): текст клаузы НЕ ссылается на номера К — говорит «оцени по КАЖДОМУ заданному критерию строго по его описанию». Поэтому когерентен и с пресетом К1–К10, и с ЛЮБЫМИ кастомными критериями репетитора (главный путь). Грамотность-исключение — условное («критерии грамотности, если такие заданы») → строгость на содержании, лояльность к подсчёту ошибок. Возвращается только для русского сочинения (gate `isEssay27`); generic-ready для физики/языков.
5. **Numeric-гейт** — короткий ответ никогда не получает клаузу (лояльность к формату записи сохраняется).
6. **Физ/мат покритериальные пресеты отложены** — пока строгий single-score балл; репетитор задаёт `grading_criteria` вручную.

### Scope
**In scope:**
- Поле `grading_discipline` + проводка в оба грейдинг-промпта.
- Контент строгости только для русского сочинения № 27.
- Forward-guard в smoke-check.

**Out of scope:**
- Контент строгости для физики/языков/математики (отдельные итерации).
- Покритериальные пресеты физики/математики.
- Изменения hint/chat/валидатора/cascade-логики.
- Любые миграции / schema-изменения.

## 4. User Stories

### Репетитор

## 5. Technical Design

### Затрагиваемые файлы
**Edge/Deno:**
- `supabase/functions/_shared/subject-rubrics/types.ts` — поле `grading_discipline?: string | null` в `SubjectRubric`.
- `supabase/functions/_shared/subject-rubrics/russian-ege.ts` — generic константа `STRICT_GRADING_DISCIPLINE` + возврат для эссе № 27 (review fix #2).
- `supabase/functions/_shared/subject-rubrics/index.ts` — коалесцинг в return `resolveSubjectRubric` с numeric-гейтом.
- `supabase/functions/homework-api/guided_ai.ts` — инжект `rubric.grading_discipline ?? ""` в `buildCheckPrompt`; **+ fix матчинга `matchTemplateEntry` (review fix #3)** — code-pass `criterionCode` против коллизии «К1»⊂«К10».
- `supabase/functions/_shared/mock-exam-prompts.ts` — инжект в `buildMockExamPart2Prompt` (no-op для физики).

**Frontend (review fix #4 → требует `deploy-sokratai`!):**
- `src/lib/gradingCriteriaPresets.ts` — `RUSSIAN_EGE_27_PRESET` дополнен `description` (byte-mirror backend `RUSSIAN_EGE_27_CRITERIA`). Кнопка пресета пишет criteria в БД → перекрывает backend-пресет → без описаний AI грейдил бы по label+max.

**Тесты/доки:** `scripts/test-criteria-templates.mjs` (grading_discipline forward-guard + #4 mirror-drift guard), `.claude/rules/40-homework-system.md` (mirror note).

### Data Model
Нет. Чисто prompt-слой + frontend-пресет. Без миграций, без новых таблиц/колонок/RPC.

### API
Нет изменений endpoint'ов. Меняется только текст системного промпта, передаваемого AI на грейдинге.

### Tone-split матрица

| Путь | Файл | `grading_discipline`? |
|---|---|---|
| Проверка/сдача ДЗ | `guided_ai.ts::buildCheckPrompt` | ДА |
| Пробник Часть 2 (грейдинг) | `mock-exam-prompts.ts::buildMockExamPart2Prompt` | ДА (провод; контент null) |
| Подсказка | `guided_ai.ts::buildHintPrompt` | НЕТ |
| Чат-обсуждение | `chat/index.ts` subjectBlock | НЕТ |
| Валидатор / fallback | `evaluateStudentAnswer`, fallback hint | поле есть, не используется |

Серверный prompt-фикс, UI не меняется. Эффект виден ученику/репетитору как более точный балл и `ai_criteria_json`. Тон `feedback` для ученика остаётся уважительным (явная строка клаузы).

## 7. Validation

### Как проверяем успех?
- Пере-сдать то же сочинение (ДЗ `c4384c4e-b438-4ea9-a42a-fa2637ed1c68`, задача `40e4e689-2987-4b54-bb9b-d8ac6fe2da75`; попытки не ограничены) после деплоя → сравнить новый `ai_criteria_json` с оценкой филолога.
- **Критерий приёмки:** балл сместился от завышенного к экспертному; расхождение с филологом по К1–К6 минимально. Если нет — **итерировать только текст клаузы** в `russian-ege.ts`.
- **Sanity:** физическое/numeric ДЗ — грейдинг не изменился (промпт байт-в-байт прежний, `grading_discipline=null`).

### Smoke check
```bash
npm run lint && npm run build && npm run smoke-check
```
Секция 9 (`test-criteria-templates.mjs`) теперь ассертит: русское эссе № 27 → непустая клауза; numeric/не-эссе русский/физика/языки → null.

### Деплой
Backend/edge-only → **Lovable синк на push** (НЕ `deploy-sokratai`). Передеплоятся `homework-api`, `chat`, `mock-exam-grade` (импортируют `_shared/subject-rubrics`). Проверка живости — `curl` функции (401 = живо, 503 = boot-crash, rule 95).

## 8. Risks & Open Questions

| Риск | Вероятность | Митигация |
|---|---|---|
| Клауза занижает балл (over-correction) | Средняя | Калибровка текста филологом на реальном сочинении; правка single-point в `russian-ege.ts` |
| Строгость утекает в добрые пути (hint/chat) | Низкая | Поле читается только в 2 грейдинг-промптах; smoke-guard на null для прочего |
| Регрессия грейдинга физики/numeric | Низкая | `grading_discipline=null` → промпт неизменен; sanity-проверка + smoke-guard |
| Противоречие К7–К10 строгости | Закрыто | Carve-out в тексте клаузы |

### Открытые вопросы
1. **Французский (до старта итерации Эмилии):** узнать её Gemini-флоу — экзамен/уровень (DELF A2/B1/B2? ЕГЭ?), критерии/формат, почему не пользуется встроенным `languages-ege.ts` (там уже есть DELF/ЕГЭ-критерии). Возможно проблема та же (нет строгости) → фикс тривиален.
