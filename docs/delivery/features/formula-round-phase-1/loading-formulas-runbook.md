# Runbook: загрузка нового раздела формул в тренажёр

**Цель:** добавить новый раздел физики (МКТ, Электродинамика, Оптика, ОГЭ-трек и т.д.) в `/trainer` за один проход, минимизировав ручной труд Vladimir и максимизировав качество пед-контента.

**Целевая аудитория:** Vladimir + Claude/Cursor агент. Workflow рассчитан на solo-оператора, который кидает скриншоты + Excel и получает PR-ready коммит.

**Канон:** пример раздела «Механика» уже в репо — `src/lib/formulaEngine/formulas.ts` (40 формул) + `src/lib/formulaEngine/questionGenerator.ts` (BUILD_RECIPES + MUTATION_LIBRARY + SITUATION_LIBRARY). Любая новая загрузка должна следовать этому канону 1-в-1.

---

## 0. Перед стартом — что должно быть готово

Vladimir готовит **три** артефакта и кидает их в одно сообщение Claude:

1. **Скриншоты учебника / методички / справочника ЕГЭ** — фотки страниц с формулами раздела. Чем больше контекста (вывод, обозначения, типичные задачи), тем лучше. Допустимы: экран iPad, фото бумажной книги, PDF-выгрузка.
2. **Excel/Google Sheets от репетитора** (если есть) — таблица с формулами в свободной форме. Колонки могут быть какие угодно: «формула», «название», «когда применять» и т.д.
3. **Имя раздела + scope:**
   - `Молекулярная физика и термодинамика` (ЕГЭ)
   - `Электродинамика` (ЕГЭ)
   - `Оптика, кванты, атом` (ЕГЭ)
   - `ОГЭ — Механика 8-9 класс` (упрощённый трек)
   - и т.д.

В сообщении Claude формулировка типа: *«Загружаем раздел Электродинамика для ЕГЭ. Вот скриншоты из Мякишева и Excel от Сергея. Действуй по runbook.»*

## 1. Архитектурные инварианты — НЕ нарушать

Перед тем как Claude начнёт писать код, он обязан перечитать эти правила (они в `.claude/rules/40-homework-system.md` секция «Тренажёр формул»):

- **Client-side формульный движок.** Никаких AI-вызовов в рантайме, никаких DB-запросов. Каталог формул статичен.
- **Three card types:** `TrueOrFalseCard` (Layer 3, мутации), `BuildFormulaCard` (Layer 2, токены), `SituationCard` (Layer 1, ситуация → выбор формулы).
- **Single source of truth для проверки** — `FormulaRoundScreen.handleAnswer`. Карточки возвращают raw answer, **не** boolean. Не переносить проверку в карточки.
- **`BuildFormulaAnswer = { numerator: string[]; denominator: string[] }`** — структурированный объект, не flat array.
- **Кириллица в индексах:** только через `\\text{}`. Пример: `s_{\\text{общ}}`, `v_{\\text{ср}}`, `t_{\\text{общ}}`. Без `\\text{}` KaTeX рендерит ломано.
- **ID-схема:** `{section}.{NN}`. Внутренние ID **никогда** не должны попадать в user-facing строки (`reasoning`, `trap`, `physicalMeaning`, мутации, ситуации). Используй человеко-читаемые отсылки или helper `formulaName(id)`.
- **`relatedFormulas`** — массив id из ТОГО ЖЕ раздела (и/или тематически близких). Используется для дистракторов в SituationCard.
- **Дистракторы в SituationCard:** `relatedFormulas` first → sameSection backfill. **Не** `shuffle(merged)`.
- **Anti-spoiler:** в условии Situation **нельзя** упоминать саму формулу или её LaTeX. Только физическая ситуация + что ищем.

## 2. Pipeline — 8 шагов

### Шаг 1. Inventory (5 мин)

Claude разбирает входные материалы:
- OCR-ит / читает скриншоты
- Парсит Excel (если есть)
- Составляет **черновой список формул** в чате — нумерованный, с raw LaTeX и кратким названием на русском
- Группирует по тематическим блокам (например, для Электродинамики: Электростатика → Постоянный ток → Магнитное поле → ЭМИ → Колебания и волны)

**Output:** табличка вида
```
№  | Тема          | Формула                  | Источник
1  | Электростатика | F = k·q1·q2/r²          | screenshot1, Мякишев §1
2  | Электростатика | E = F/q                 | screenshot1
...
```

**Stop point:** Claude показывает inventory Vladimir и спрашивает: *«Это полный список? Что-то добавить / убрать / переименовать?»* Никакого кода до подтверждения.

### Шаг 2. ID-нейминг и нумерация

### Шаг 3. Заполнение `formulas.ts` — полная Formula-shape

Для каждой формулы Claude генерит **черновик** со всеми полями `Formula` interface (см. `src/lib/formulaEngine/types.ts`):

```ts
{
  id: 'el.01',
  section: 'Электростатика',           // user-facing label
  topic: 'Закон Кулона',                // подтема
  name: 'Закон Кулона',                 // короткое название (для feedback header)
  buildTitle: 'Сила взаимодействия точечных зарядов',  // конкретно, без жаргона
  formula: 'F = k \\frac{|q_1 q_2|}{r^2}',  // LaTeX, escape для TS
  formulaPlain: 'F = k|q₁q₂|/r²',          // plain для fallback
  variables: [
    { symbol: 'F', name: 'сила взаимодействия', unit: 'Н' },
    { symbol: 'q_1, q_2', name: 'заряды', unit: 'Кл' },
    { symbol: 'r', name: 'расстояние между зарядами', unit: 'м' },
    { symbol: 'k', name: 'коэффициент', unit: 'Н·м²/Кл²' },
  ],
  physicalMeaning: 'Сила между точечными зарядами прямо пропорциональна произведению зарядов и обратно пропорциональна квадрату расстояния. Без internal ID-ссылок!',
  proportionality: {
    direct: ['q_1 → F', 'q_2 → F'],
    inverse: ['r² → F'],
  },
  dimensions: '[Н] = [Н·м²/Кл²]·[Кл²]/[м²]',
  derivedFrom: 'Эмпирический закон (опыт Кулона)',  // или название более общего принципа
  whenToUse: [
    'Точечные неподвижные заряды в вакууме',
    'Дано: q₁, q₂, r — найти F',
    'Знак силы определяется знаками зарядов (отталкивание/притяжение)',
  ],
  commonMistakes: [
    'Забывают модули — закон даёт величину силы, направление определяется отдельно',
    'Путают k и ε₀ (k = 1/(4πε₀))',
  ],
  relatedFormulas: ['el.02', 'el.03'],  // 2-4 ближайших формулы из того же раздела
  difficulty: 1,  // 1=база, 2=средне, 3=сложно
}
```

**Правила качества для AI-черновика:**
- `physicalMeaning` — 1-2 предложения, объясняет суть. **Никаких** «kin.03», «эта формула» — пиши как преподаватель.
- `whenToUse` — минимум 3 буллета: триггеры в условии задачи. Это не строгая теория, а маркеры распознавания.
- `commonMistakes` — реальные ошибки школьников (знаки, единицы, путаница похожих формул). Если методист/Vladimir знает конкретные ловушки — впиши их в первую очередь.
- `relatedFormulas` — обязательно ≥ 2 id, иначе SituationCard будет генерить плохие дистракторы.
- `buildTitle` — **обязательно**, чтобы BuildFormulaCard не падал на fallback `formula.name`. Конкретно описывает физический смысл, без LaTeX и без жаргона.

### Шаг 4. `questionGenerator.ts` — BUILD_RECIPES

Для **каждой** формулы добавить запись в `BUILD_RECIPES`:

```ts
'el.01': {
  displayFormula: 'F = k \\frac{|q_1 q_2|}{r^2}',
  numeratorTokens: ['k', '|q_1 q_2|'],
  denominatorTokens: ['r^2'],
},
```

**Правила токенизации:**
- Токены — **атомарные блоки**, которые ученик собирает мышкой. Не дроби внутри дробей.
- Если формула — не дробь, `denominatorTokens: []`.
- Если формула — сумма (например, `s = v₀t + at²/2` = `kin.04`), сплющить в numerator: `['v_0', 't', 'a', 't^2']`, denominator `['2']`. Это компромисс — см. CLAUDE.md «известное ограничение builder-а».
- Кириллица в индексах — **только** через `\\text{}` (см. инвариант выше).
- Использовать LaTeX-нотацию консистентно: `q_1`, `r^2`, `\\sqrt{...}`, `\\frac{...}{...}`.

### Шаг 5. `MUTATION_LIBRARY` — для TrueOrFalseCard

Для каждой формулы — **2-3 мутации** (искажённые версии формулы для проверки «верно/неверно»):

```ts
'el.01': [
  {
    type: 'wrong_power',
    latex: 'F = k \\frac{|q_1 q_2|}{r}',
    hint: 'Сила обратно пропорциональна КВАДРАТУ расстояния, а не самому r',
  },
  {
    type: 'drop_coefficient',
    latex: 'F = \\frac{|q_1 q_2|}{r^2}',
    hint: 'Без коэффициента k формула не имеет правильной размерности',
  },
  {
    type: 'swap_fraction',
    latex: 'F = k \\frac{r^2}{|q_1 q_2|}',
    hint: 'Перевёрнута дробь — заряды должны быть в числителе',
  },
],
```

**Типы мутаций (фиксированный enum):**
- `swap_fraction` — перевёрнута дробь (числитель ↔ знаменатель)
- `drop_coefficient` — потерян множитель / коэффициент
- `wrong_power` — неверная степень
- `swap_variable` — подменена переменная (например, `v` вместо `v_0`)

**Правила для hint:**
- Hint — **подсказка после ответа**, объясняет в чём ошибка мутации.
- Без префиксов `Запомни:`, `Триггер:`, `Ловушка:` (мы их специально вырезали в Phase 4).
- Без internal ID. Если ссылаешься на другую формулу — по человеческому названию.
- 1-2 предложения, без воды.

---

### Шаг 6. `SITUATION_LIBRARY` — для SituationCard

Для каждой формулы — **1-2 ситуации**:

```ts
'el.01': [
  {
    prompt: 'Два маленьких заряженных шарика висят на расстоянии 5 см друг от друга. Известны заряды и расстояние. Какую формулу применить, чтобы найти силу между ними?',
    rationale: 'Точечные заряды + расстояние + ищем силу = закон Кулона',
  },
],
```

**Anti-spoiler правила (КРИТИЧНО):**
- В `prompt` **запрещено** упоминать саму формулу или её LaTeX (`F = kq₁q₂/r²` — нельзя!)
- Запрещено упоминать имя формулы (`закон Кулона` в условии — нельзя, это и есть ответ)
- Можно: физическая ситуация, что дано, что ищем
- 2-3 предложения максимум, без художественной воды

**`rationale`** — объяснение для feedback после ответа, почему именно эта формула. Не должен дублировать `physicalMeaning`, должен фокусироваться на маркерах распознавания из `prompt`.

### Шаг 8. Валидация и self-test

**Перед коммитом** Claude обязан:

1. **`npx tsc --noEmit`** — TypeScript должен быть зелёным. (Vite build может падать на Bus error в sandbox-e — это окружение, не код.)
2. **Self-check каталога:**
   - Каждая формула имеет ВСЕ обязательные поля (`id`, `section`, `topic`, `name`, `buildTitle`, `formula`, `formulaPlain`, `variables`, `physicalMeaning`, `proportionality`, `dimensions`, `derivedFrom`, `whenToUse`, `commonMistakes`, `relatedFormulas`, `difficulty`)
   - Каждая формула имеет запись в `BUILD_RECIPES`
   - Каждая формула имеет 2-3 записи в `MUTATION_LIBRARY`
   - Каждая формула имеет ≥ 1 запись в `SITUATION_LIBRARY`
   - `relatedFormulas` ссылается только на существующие id из ТОГО ЖЕ раздела
3. **Anti-spoiler scan:** grep по `SITUATION_LIBRARY.prompt` на упоминания LaTeX-команд (`\\frac`, `\\sqrt`, `^`, `_`) и имён формул (`закон`, `принцип`, `теорема`). Если есть — переписать.
4. **ID-leak scan:** grep по всем user-facing полям (`physicalMeaning`, `commonMistakes`, `whenToUse`, mutation `hint`, situation `rationale`) на pattern `\b(kin|dyn|cons|stat|hydro|el|mag|...)\.[0-9]+\b`. Должно быть пусто.
5. **Cyrillic-subscript scan:** grep по `BUILD_RECIPES` на pattern `_[а-яё]` без `\\text{` рядом. Должно быть пусто.

Только после всех 5 проверок — отчитаться Vladimir и предложить команду коммита.

## 5. ОГЭ-трек — что отличается

Параллельный набор для 8-9 класса:
- **Префикс** id отдельный: `oge_kin`, `oge_dyn`, `oge_el`, и т.д. (или один общий `oge_*`)
- **Меньше формул** в разделе (программа ОГЭ ýже)
- **Упрощённые формулировки** в `physicalMeaning` и `whenToUse` — язык 8-классника, без производных, без векторного анализа
- **Мнемоники проще** — больше визуальных аналогий, меньше формальной математики
- **Отдельная секция в `TrainerPage`** — две группы кнопок: «ЕГЭ» и «ОГЭ», между ними визуальный разделитель
- **Не пересекать** ОГЭ и ЕГЭ формулы в `relatedFormulas` — это разные тренировочные пулы

## 6. Anti-patterns — что Claude НЕ должен делать

- ❌ Гуглить или искать формулы в Web — все источники приходят от Vladimir в виде материалов
- ❌ Изобретать формулы, которых нет в скриншотах
- ❌ Писать `physicalMeaning` копированием Wikipedia — требуется педагогический язык
- ❌ Пропускать поля Formula-shape «потому что неважно» — все 14 полей обязательны
- ❌ Использовать `framer-motion` или другие тяжёлые либы (ban из `performance.md`)
- ❌ Создавать новые route или top-level страницы — только дополнять `TrainerPage` и `formulas.ts`
- ❌ Менять `FormulaRoundScreen.handleAnswer` — single source of truth, не трогать
- ❌ Делать commit без `npx tsc --noEmit` зелёного

## 7. Известные ограничения (не баг, не править в этой итерации)

- **Builder сложных сумм** (`a + b/c`): BUILD_RECIPE флэтит токены в numerator/denominator. Для формул типа `s = v₀t + at²/2` это рабочий компромисс, но не отражает структуру суммы. При расширении на электродинамику будет много таких — принимаем как есть.
- **Vite build Bus error** в sandbox-e — environmental, игнорируем. Достаточно `npx tsc --noEmit`.
- **`buildTitle`** не у всех формул механики — fallback на `name`. Новые разделы должны иметь `buildTitle` у 100% формул.
