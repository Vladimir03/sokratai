## Section 0 — Job Context

**Core Job (AJTBD):** R1-2 «Понять, что усвоил каждый ученик после ДЗ — за 30 секунд, не открывая каждую переписку».

Source-of-truth: `docs/discovery/product/tutor-ai-agents/15-backlog-of-jtbd-scenarios-sokrat.md`.

Связанные jobs:
- R1-3 «Сравнить учеников между собой» (нужны сопоставимые числа: балл, подсказки, время).
- R3 «Дотянуть отстающего» (число подсказок и время — proxy для сигнала «застрял/не понимает»).

**Wedge alignment:** репетитор физики ЕГЭ/ОГЭ открывает результаты ДЗ перед занятием. Сейчас он видит хитмап клеток, но **в строке ученика нет агрегата** — приходится складывать клетки глазами. Это ломает обещание «обзор за 30 секунд».

---

## 1. Problem

В текущем `TutorHomeworkDetail` и Results-хитмапе для каждого ученика показаны только баллы по отдельным задачам. **Нет** трёх ключевых агрегатов в правой части строки:

1. **Сумма баллов** `score / max` (например, `9/11`).
2. **Общее число подсказок**, использованных в guided-чате по всему ДЗ.
3. **Общее время**, затраченное на ДЗ (от первого сообщения в треде до последнего, либо до явного «завершить»).

Из-за этого:

- Репетитор не может за один взгляд отсортировать «кто молодец / кто плывёт».
- Невозможно увидеть аномалию «решил всё, но за 2 часа и с 12 подсказками» — это сигнал «списал у GPT» или «не понял, но добил».
- Сравнение учеников между собой требует ручного суммирования.

В моке `HomeworkResultsV2Mock.jsx` эти три колонки уже нарисованы (Балл / Лампочка / Время) и являются ключевым визуальным акцентом справа. Цель PRD — закрепить требования и превратить мок в production.

---

## 2. Goal

После релиза репетитор, открывая `/tutor/homework/:id`, видит для каждого ученика в одной строке: распределение баллов по задачам (хитмап) **и** три агрегата справа — сумма баллов, подсказки, время.

**Primary success metric:**
- Время до первого осмысленного решения «с кем поговорить на занятии» ≤ 30 секунд (qual: Женя на пилоте).
- ≥80% репетиторов-пилотов в интервью говорят, что «колонка время» / «колонка подсказки» поменяла их интерпретацию хотя бы одного ученика.

---

## 3. Users and stories

- **Репетитор-пилот (Женя, физика ЕГЭ).** «Открываю ДЗ перед занятием, хочу увидеть кто провалил, кто перестарался, кто списал — без кликов внутрь треда».
- **Репетитор олимпиадник (Егор).** «Хочу видеть, кто решил быстро без подсказок — это сильные кандидаты в продвинутую группу».
- **Out of scope:** ученик и родитель (отдельный экран, не этот PRD).

---

## 4. Solution overview

В строке ученика хитмапа / детальной страницы ДЗ — три новых правых колонки:

| Колонка | Источник | Формат |
|---|---|---|
| **Балл** | Σ `final_score` по всем `task_states` ученика / Σ `max_score` по всем задачам ДЗ | `9/11` (полужирный slate-900) или `—` если не приступал |
| **Подсказки** | Σ `hint_count` по всем `task_states` ученика | число + иконка `Lightbulb`; чип `bg-amber-100` если ≥ `HINT_OVERUSE_THRESHOLD` |
| **Время** | `last_message_at − first_message_at` thread'а ученика по этому ДЗ, округлено до минут | `47 мин` или `—` |

Под последней строкой таблицы — итоговая строка «Среднее по задаче» (уже есть в моке).

**Без сортировки на v1** — клики по заголовкам колонок добавляются во вторую итерацию (Parking Lot).

## 5. Out of scope (v1)

- Сортировка по любой из 3 колонок (P1).
- История попыток (кол-во отправок) — отдельный сигнал, не агрегат (P2).
- Сравнение со средним по группе (`+5 мин против среднего`) — P2.
- Экспорт в Excel/PDF (P2).
- Колонка «время» для учеников, которые ещё в процессе (показываем `— в процессе` без числа).

## 6. Risks

| # | Риск | Митигация |
|---|---|---|
| R1 | `time = last − first` обманчиво при паузах (ученик закрыл и пришёл через сутки) | В v1 принимаем как proxy; в P1 — switch на «активное время» (sum коротких сессий) |
| R2 | Hint count в БД может быть рассинхронизован между `task_states.hint_count` и фактическими сообщениями `kind='hint'` | Источник истины — `task_states.hint_count` (как в Results v2). Backfill отдельной миграцией если найдётся drift |
| R3 | Колонки не помещаются на iPhone SE | Sticky колонка имени + горизонтальный скролл, последние 3 колонки — всегда в скролле, не sticky |
| R4 | «Время» для не сдавших — `null`. Может смутить «сдал, но 0 мин» если thread пустой | Минимум 1 мин на отображение, если submitted |

## 7. Open questions

1. Время — wall-clock (last − first) или активное (sum интервалов <5 мин)? **→ предлагаю v1 wall-clock как простое решение, P1 — switch.**
2. Куда писать `total_time_min` — вычислять на лету в `handleGetResults` или денормализовать в `homework_tutor_threads.total_time_min`? **→ v1 на лету (мы и так делаем агрегаты в этом эндпоинте), денормализация если станет тормозить.**
3. Колонка «Балл» дублирует число в шапке-сводке — это норм? **→ Да, шапка = по группе, колонка = по ученику.**

## 9. Definition of done

- Три колонки видны в `TutorHomeworkDetail` (`/tutor/homework/:id`) для каждого ученика.
- Клик по строке ученика по-прежнему раскрывает drill-down (не ломаем Results v2 TASK-6).
- Цифры консистентны: сумма по строке = сумме клеток той же строки.
- Time для не сдавших = `—`, для сдавших ≥ 1 мин.
- Hint chip (≥ threshold) визуально совпадает с тем, что показывает action block в шапке.
- npm run lint && build && smoke-check зелёный.
- Manual smoke на iPhone SE 375px (горизонтальный скролл).
