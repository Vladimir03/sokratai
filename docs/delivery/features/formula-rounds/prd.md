## 1. Job Context

- **Core Job**: R4 — Сохранение контроля и качества при масштабировании
- **Sub-job**: R4-1 — Быстро собрать качественное ДЗ по теме урока
- **Вторичные Jobs**:
  - S1-2 — Выбрать правильный подход к решению
  - S1-4 — Понять ошибку и не повторить её в следующей задаче
  - S2-4 — Увидеть свой прогресс и почувствовать рост
- **Segment**: репетиторы физики ЕГЭ/ОГЭ с 10+ учениками / мини-группами + их ученики
- **Wedge alignment**: Да — добавляет в homework flow новый тип практики по теме урока, который репетитор может назначить так же быстро, как задачу
- **Pilot impact**: усиливает ежедневную практику между уроками и делает homework workflow ценнее для связки "репетитор назначает → ученик проходит → репетитор видит прогресс"

---

## 2. Problem

Сейчас в homework flow репетитор может назначать задачи, материалы и guided-chat взаимодействие, но не может дать лёгкий, короткий, измеримый блок на повторение формул по теме урока.

Из-за этого:

- ежедневная практика между уроками опирается только на полноценные задачи, которые для части учеников слишком "тяжёлые" как первый шаг;
- у ученика нет короткого цикла "вспомнить формулу → понять, когда она применяется → увидеть микро-прогресс";
- у репетитора нет отдельной видимости: какие формулы ученик узнаёт, путает или системно не применяет;
- повторение формул происходит вне системы: списки, скриншоты, устные просьбы "повтори кинематику", что не даёт ни completion signal, ни продуктовой привычки.

**Текущий workaround**:

- репетитор просто добавляет больше задач в ДЗ;
- отдельно просит "выучить формулы";
- ученик сам открывает шпаргалки, PDF или чужие сервисы;
- прогресс по формулам не фиксируется и не виден в homework workflow.

**Суть боли**: между "сложной задачей" и "ничего не делать" нет встроенного micro-drill слоя, который можно быстро назначить по теме урока и потом увидеть в результатах.

---

## 3. Solution

Добавить в homework flow новый тип артефакта: **formula round**.

Репетитор при создании ДЗ может прикрепить к assignment не только задачи, но и один назначаемый раунд тренажёра формул по теме. В Phase 1 это будет **один шаблонный round по кинематике** на базе существующей формульной базы (`docs/mechanics-formulas.json`) и правил из GDD (`docs/SokratAI_physics_game-design-document.md`).

**Реализация тренажёра определяется GDD** — документ описывает конкретные механики, которые SPEC должен специфицировать: три типа заданий из 12 возможных (§4.1 Правда/ложь, §4.5 Собери формулу, §4.8 Ситуация→Формула), алгоритм генерации мутаций для дистракторов (§4.1), систему feedback после каждого ответа (§7), структуру раунда (§2.2) и порядок слоёв (§3.2).

Student experience:

- в [StudentHomeworkDetail](/c:/Users/kamch/sokratai/src/pages/StudentHomeworkDetail.tsx) появляется отдельная карточка блока "Тренажёр формул";
- ученик открывает раунд из ДЗ, проходит 10 заданий за 3-5 минут;
- получает мгновенный feedback, объяснение и итоговый экран с ошибками;
- может повторить ошибки.

Tutor experience:

- в [TutorHomeworkCreate](/c:/Users/kamch/sokratai/src/pages/tutor/TutorHomeworkCreate.tsx) репетитор включает formula round как часть ДЗ;
- в tutor results / detail видит completion, score и слабые формулы ученика;
- может использовать раунд как ежедневную micro-practice по теме урока, не создавая отдельный продуктовый контур.

### Продуктовое решение по входу

Для **v1** первый вход делаем **внутри ДЗ**, а не отдельным top-level экраном "Тренажёр".

Почему это правильнее для Phase 1:

- это усиливает существующий wedge вместо создания второго student-side продукта;
- сохраняет job framing: репетитор назначает practice после урока, а не "открывает игру";
- упрощает completion tracking и tutor visibility в текущем homework flow;
- поддерживает одну главную работу экрана и avoids chat/game drift.

Отдельный экран "Тренажёр" рассматривается как **LATER / Phase 2+** для самостоятельного повторения и habit loop.

## 4. Scope

### IN (делаем)

- Новый homework-linked артефакт: `formula_round`
- Добавление formula round в assignment из [TutorHomeworkCreate](/c:/Users/kamch/sokratai/src/pages/tutor/TutorHomeworkCreate.tsx)
- Phase 1 content scope: только **Кинематика**
- Контентная база Phase 1: 12 формул кинематики из `docs/mechanics-formulas.json`
- Три типа заданий из GDD:
  - `Правда или ложь`
  - `Собери формулу`
  - `Ситуация → Формула`
- Один round = 10 заданий
- 3 жизни на round
- Мгновенный feedback после каждого ответа
- Итоговый экран раунда с:
  - score / % правильных
  - списком проблемных формул
  - CTA `Повторить ошибки`
- Базовый student progress по формулам и раундам
- Tutor visibility с первого дня:
  - прошёл / не прошёл
  - best score
  - last played
  - top weak formulas + слой проблемы для каждой (текстом: «путает структуру» / «не собирает» / «не узнаёт в задаче»)
- Student feedback: score (N/10), % правильных, проблемные формулы. Без XP — мотивация через score improvement при повторах
- Student UX lives внутри existing homework surface, а не в отдельном приложении
- Web-first реализация в существующем student homework flow
- Round назначается как один шаблон «Кинематика» (без подуровней) — система сама подбирает 10 заданий из 12 формул
- Formula round входит в общий % completion ДЗ — без round ДЗ не считается выполненным на 100%

### OUT (не делаем)

- Отдельный top-level экран "Тренажёр"
- Telegram-first gameplay
- Вся механика целиком (кроме кинематики)
- Остальные 9 типов заданий из full GDD
- Boss-раунды
- Карта прогресса по разделам
- Streak / заморозка streak / daily rewards
- Spaced repetition / adaptive engine уровня full vision
- Связка результатов тренажёра с guided chat / AI-подсказками
- XP-система и мета-экономика (уровни, бонусы за серии)
- Генерация новых формул или новых разделов через AI

### LATER (потом)

- Остальные разделы механики: динамика, законы сохранения, статика, колебания и волны
- Остальные 9 task types из GDD
- Отдельный экран "Тренажёр"
- XP-система с накоплением и уровнями ученика (имеет смысл при standalone экране)
- Streak / карта прогресса / boss-раунды
- Spaced repetition и адаптивная подача заданий
- Назначение раундов по слабым темам из результатов ДЗ / guided chat
- Рекомендации "какой round назначить дальше"

## 5. User Stories

**Репетитор:**

- После урока по кинематике я создаю ДЗ и добавляю к задачам короткий formula round, чтобы ученик повторил базовые формулы между уроками.
- Я хочу назначить round по теме урока, не выходя в отдельный продуктовый контур и не собирая контент вручную.
- Я хочу открыть результаты ученика и сразу увидеть: прошёл ли он round, с каким score и какие формулы он путает.

**Ученик:**

- Я открываю ДЗ и вижу короткий тренажёр по формулам темы, который можно пройти за несколько минут, а не только длинные задачи.
- Я хочу быстро понять, какую формулу я путаю и почему ответ неверный.
- Я хочу видеть, что у меня есть микро-прогресс, даже если я ещё не идеально решаю полные задачи.

**Связка tutor + student:**

- Репетитор назначает round внутри ДЗ, ученик проходит его в том же homework flow, а результат сразу становится частью общей картины выполнения домашки.

## 6. Success Criteria

### Leading (3-7 дней после релиза)

- **Adoption**: пилотный репетитор добавляет formula round минимум в 3 assignment-а по кинематике в первую неделю
- **Student activation**: минимум 60% учеников, которым назначили round, запускают его в течение 24 часов после выдачи ДЗ
- **Completion**: минимум 40% запустивших учеников завершают полный round из 10 заданий
- **Session fit**: median time to complete одного round-а ≤ 5 минут
- **Tutor visibility**: репетитор минимум 1 раз в неделю открывает результаты по formula round и использует их для review

### Lagging (2-4 недели)

- **Habit signal**: у активных учеников появляется повторяемый паттерн ежедневных раундов, минимум 3 активных дня в неделю
- **Workflow retention**: репетитор продолжает назначать formula round как supporting layer внутри homework flow, а не только "ради теста новой штуки"
- **Qualitative signal**: репетитор подтверждает, что round помогает поддерживать практику между уроками и не воспринимается как "лишняя игрушка"
- **Learning signal**: у репетитора появляется полезная картина по слабым формулам, а не только "сделал / не сделал ДЗ"

## 8. Risks

| Риск | Вероятность | Mitigation |
|---|---|---|
| Фича расползётся в отдельный student-game продукт и ослабит wedge | Средняя | Жёстко держать Phase 1 внутри homework flow; отдельный экран вынести в LATER |
| Репетитор не будет назначать round, если это усложняет конструктор ДЗ | Средняя | Добавление в assignment должно быть шаблонным и занимать 1-2 клика |
| Ученик воспримет round как необязательный и будет игнорировать | Средняя | Встроить round прямо в ДЗ и показывать visible completion state |
| Слишком бедная аналитика не даст репетитору ценности | Средняя | С первого дня показать хотя бы completion + best score + weak formulas |
| Слишком широкая геймификация затянет delivery и убьёт Phase 1 | Высокая | Ограничить Phase 1 тремя типами заданий, кинематикой и одним homework-linked entry point |
| Формульные дистракторы/объяснения будут ощущаться "школьно-неряшливо" и снижать доверие | Средняя | Брать дистракторы и when-to-use из `mechanics-formulas.json`, не генерировать на лету без правил |
