## T1 — Добавление ученика по имени (контакт опционален)
**Reuse:** `AddStudentDialog.tsx`, `tutor-manual-add-student`, `tutor_students`, `tutor_get_invite_code`.
**Дельта:** имя обязательно, контакт (email/Telegram) опционален; bulk-поле «список имён»; снять обязательность контакта в edge; **обновить rule 60** синхронно.
**AC:** репетитор заводит ученика, введя только имя (edge не требует контакта); добавление N учеников вставкой списка; плейсхолдер пишется с контактом `NULL`. Edge-ошибки — flat `{error, code}` (rule 97).

## T2 — Плейсхолдер: ДЗ копится до подключения
**Reuse:** конструктор ДЗ + assign-эндпоинты `homework-api` (rule 40).
**Дельта:** разрешить assign/сборку ДЗ на плейсхолдер без канала; `provisionGuidedThread` как сейчас.
**AC:** репетитор собирает и назначает ДЗ ученику, который ещё не подключился; ДЗ ждёт в его привязке.

## T3 — Share-gate «Подключить ученика» (одноразовый)
**Reuse:** `getTutorInvitePreviewLink`, `tutor_get_invite_code`, статус активации (registration-onboarding).
**Дельта:** новый `ConnectStudentSheet.tsx` (QR + ссылка; вторым планом «отправить на email/Telegram» = capture канал + отправка приглашения+ДЗ). Backend assign/notify возвращает `students_without_channel` (зеркало `students_without_telegram_names`) → фронт открывает sheet. **Без отдельного «кода» в UI.**
**AC:** при первой отправке ДЗ ученику без канала открывается sheet; любой способ захватывает канал при claim; «отправить на email/Telegram» доставляет приглашение+ДЗ; **после подключения sheet не появляется** (отправка молча).

## T4 — claim → беспарольная сессия → «ты подключён» + превью
**Reuse:** `claim-invite`, `email-verify` (минтинг сессии, rule 96), `InvitePage.tsx`, резолвер первой задачи, problem screen.
**Дельта:** claim минтит сессию server-side (per-student токен, авто-привязка без апрува); экран «вы подключены к {репетитор}» + превью задания; кнопка ведёт на регистрацию (T6), затем — задача. Видимая привязка (фикс тихого claim, F3).
**AC:** ученик по ссылке/QR видит «подключён» + превью; claim-токен **одноразовый/короткоживущий** (не вечный bearer); ссылку не хард-блокируем (сессия минтится, гейт регистрации в UI).

## T5 — Persistent session (повседневный вход)
**Reuse:** Supabase session (default), `AuthGuard` (`INITIAL_SESSION`-guard).
**Дельта:** убедиться, что refresh-token персистится → повторное открытие sokratai.ru = автологин без ввода.
**AC:** ученик возвращается на том же устройстве и сразу внутри, без ссылки/пароля.

## T6 — Обязательная регистрация ДО задачи + автосейв прогресса
**Reuse:** `InvitePage.tsx`, профиль ученика, `HomeworkProblem`.
**Дельта:** между «ты подключён» и задачей — экран регистрации: email (подставлен репетитором, если есть; иначе вводит ученик) + пароль (ученик задаёт). **Без «Позже», без верификации.** Причина мягкая («чтобы не потерять прогресс»). Регистрация = доустановка пароля поверх активной сессии. **Экран «сохрани доступ» после ДЗ — убран.** Автосейв прогресса по токену в `HomeworkProblem` (прогресс не теряется при незавершённой задаче).
**AC:** ученик не попадает в задачу, не задав пароль (UI-гейт); email подставлен репетитором, если был; нет «Позже»/верификации; прогресс сохраняется по токену даже при незавершённой задаче.

## T7 — Вход с нового устройства + OTP
**Reuse:** Supabase OTP / magic-link или edge по паттерну `email-verify`.
**Дельта:** экран входа ученика (email + пароль) + «войти по коду» (OTP/magic-link на email, короткоживущий); «забыли пароль»; фолбэк-текст «нет пароля и почты — попроси у репетитора новую ссылку». **Только email, без телефона.**
**AC:** с нового устройства ученик входит email+паролем или кодом на почту; без канала — честный фолбэк на новую ссылку.

## T8 — Nudge на карточке ученика (репетитор)
**Reuse:** карточка ученика, статус активации.
**Дельта:** «не подключился» + ненавязчивый «добавьте email или Telegram, чтобы ученик мог входить с любого устройства».
**AC:** на не-подключённом ученике виден статус + nudge с действием «добавить email/Telegram».

## T9 — Телеметрия-воронка (двусторонняя)
**Reuse:** паттерн `homeworkTelemetry.ts`.
**Дельта:** таблица `analytics_events` (Supabase) + серверные события §7.2 (added, invite_generated, invite_claimed, homework_sent_to_student, student_received_and_opened, student_first_login, student_first_homework_opened, student_first_submission). PII-free. **Не** Yandex Metrica.
**AC:** события пишутся с `user_id` и джойнятся с `tutor_students`/homework; видно cross-side событие «ученик получил и открыл».

## T10 — Регрессии, валидация, деплой
