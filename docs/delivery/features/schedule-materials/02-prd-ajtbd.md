## 2. Четыре силы прогресса (по Вадиму)

- **Push (что толкает от текущего):** записи разбросаны по Google Drive, ученикам шлёт через ТГ — «так себе», нет структуры `[L1541]`. Споры с родителями про ценность абонемента `[L1563]`.
- **Pull (что тянет к новому):** «Сократ как основная платформа», перевёрнутый класс, co-builder mindset; «ученик заходит в Среду → видит тему + запись + конспект + ДЗ» `[L1609]`.
- **Anxiety (тревога перехода):** «уже есть ученик для миграции, но пока некуда» `[L2214-2216]` — блок-фактор именно отсутствие student-facing места.
- **Habit (что держит на старом):** Drive + ТГ уже работают «как-нибудь»; миграция требует, чтобы новое было заметно удобнее, иначе откат к ТГ.

**Вывод для приоритизации:** killer-job = **хранение и доставка**, не проверка `[L1603]`. Значит ядро ценности — (а) репетитор без трения крепит материалы к занятию, (б) ученик находит их в одном предсказуемом месте. Всё, что не обслуживает эти два момента, — вне v1.

## 3a. Финальная единая модель ДЗ/занятия + уточнения (подтверждено 2026-06-02, 5/5)

После конкурентного разбора (`03-competitive-ux.md`) Владимир подтвердил 5 решений:

**Модель ДЗ ↔ занятия:**
- ДЗ — самостоятельный объект (конструктор не меняется); существует **с занятием или без**.
- Занятие ссылается на ДЗ мягкой связью `homework_ref`; **1:1 в v1** (одно ДЗ ↔ один урок; many-to-one — LATER).
- Связь = ссылка, не владение → ничего не дублируется, ничего не ломается у тех, кто не ведёт расписание.

**Student-навигация: НОВАЯ вкладка «Занятия» рядом с существующей «Домашка»** (решение 2026-06-02 после ревью реальной навигации — НЕ сегмент-контрол в одном экране):
- **Терминология:** рабочее имя «Среда» → финальное имя вкладки **«Занятия»** (выбрано 2026-06-02; «Среда» отвергнуто — читается как день недели).
- **«Занятия»** — НОВАЯ top-level вкладка, **стартовый экран/лендинг ученика**: компактный список занятий **в стиле существующей «Домашки»** (группы по датам Сегодня / На этой неделе / Прошедшие; иконка-предмет; «предмет · репетитор · время»; чипы материалов **Запись/Конспект** + статус ДЗ; шеврон; метрики в шапке). Якорь «сегодня»; occurrence показывается **даже без материалов** («материалов пока нет») — расписание = ось.
- **«Домашка»** — **оставляем как есть, reuse UI 1:1.** НЕ удаляем, НЕ редиректим. Прежнее решение «убрать вкладку ДЗ» **снято**.
- **Кросс-линк (P0): чип ДЗ на карточке занятия = прямой deep-link в ОДИН КЛИК** на экран этой домашки (guided-homework problem screen), без промежуточных страниц. Обратно: у задачи в «Домашке» подпись «к занятию: …» ведёт на занятие.
- **Единый источник:** одно ДЗ показывается чипом на занятии и строкой в «Домашке» — без дублей данных (связь = reverse-lookup `tutor_lesson_materials.homework_assignment_id`).
- Standalone-ДЗ остаются в «Домашке» — ничего не теряется.
- Грациозная деградация: нет занятий → «Занятия» пусто с подсказкой, «Домашка» полностью рабочая.

**Уточнения из ресёрча (IA-эталон — Google Classroom: занятие = «тема»-карточка с 3 слотами + явные статусы ДЗ + ≤3 «ближайших due» сверху + сворачивание):**
- **Запись = generic URL** (Drive/Яндекс.Диск/VK/YouTube/любой) — РФ-риск, см. выше.
- **Создание ДЗ из drawer** (не только выбор) — открывает конструктор prefilled (предмет+получатели из занятия), на сохранении авто-привязка к занятию. Один путь создания.
- **Нудж «Добавить запись?»** при «Отметить проведённым» — **в Phase 1** (тонко: post-success подсказка → тот же drawer; логику завершения не трогаем; `TutorSchedule.tsx` high-risk → plan mode).
- **Уведомление = дайджест per-lesson** «Материалы к Уроку N готовы» (батч на «Готово»), deep-link на карточку; Telegram не единственный канал (web-push/email fallback).
- **Habit (P1):** weekly-streak на осмысленное действие (сдал ДЗ к занятию) + нудж непросмотренной записи. Не очки/подарки (rule 90 «серьёзное рабочее место»).

## 4. Scope v1

> Дельты единой модели — см. §3a. Ниже базовый scope v1, обновлённый этими решениями.

### IN
1. **Крепление материалов к занятию** (репетитор): drawer «Материалы занятия» из календаря «Расписание».
   - Drive-ссылка (URL + подпись).
   - PDF-конспект (upload в новый Storage bucket, dual-format ref).
   - Связь с существующим ДЗ (`homework_ref`).
   - Для групповой сессии — крепление на уровне `group_session_id`.
2. **Student-facing «Среда»** (ученик): лента «мои занятия» (mobile-first) — дата + тема + чипы материалов + переход в ДЗ.
3. **Вкладка «Занятия»** на странице ученика `/tutor/students/:id` (сестринская к «Прогресс»/«Отчёт») — репетитор видит занятия+материалы конкретного ученика.
4. **Уведомление** ученику при добавлении материала (существующий каскад).

### OUT (v1)
- Google Drive API/OAuth, embed-превью, синк (только URL).
- Видеохостинг (только ссылка на Drive).
- Per-student индивидуальный доступ к материалам внутри группы.
- Сохранение прогресса ДЗ при паузе (Cloud.Text-баг `[L1660]`) — отдельная фича.
- Billing/абонементы.

### LATER
- **Мини-комьюнити группы** (групповой чат/лента совместного развития) — кандидат R5-3, отдельная большая фича.
- **Cross-group видимость**: ученик видит чужие занятия по теме и «присоединяется» `[L1686]`.
- Общий каталог записей по темам.

## 5. UX-флоу

### 5.1 Репетитор — «после урока залил запись» (R4-2/R4-3)
1. Вкладка **«Расписание»** → клик по блоку прошедшего занятия.
2. Открывается drawer **«Материалы занятия»** (sheet справа/снизу, mobile-aware):
   - Заголовок: тема (`subject`) + дата/время занятия (read-only).
   - Блок **Запись**: поле URL + подпись → кнопка «Добавить ссылку».
   - Блок **Конспект**: drag-and-drop / выбор PDF → upload с прогрессом.
   - Блок **Домашка**: «Выбрать ДЗ» → пикер существующих `homework_tutor_assignments` ученика/группы.
   - Список добавленного с возможностью удалить/переименовать; reorder — LATER.
   - Тумблер/подсказка: «Ученик получит уведомление» (каскад).
3. Закрытие drawer → материалы видны ученику и в его «Среде», и в `Расписании`, и во вкладке «Занятия».

**Групповое занятие:** из `GroupDetailsDialog` тот же drawer, но материалы крепятся к `group_session_id` → видны всем участникам.

**Принципы UX:** один primary CTA на состояние; не трогаем create-форму урока; переиспользуем паттерн материалов ДЗ (`HWMaterialsSection`/`MaterialCard`) и attachment-хелперы; iOS: `font-size:16px`, `touch-action:manipulation`, 100dvh (rule 80).

### 5.2 Ученик — «Среда» (S3)
1. Новый student-route (напр. `/student/schedule`), под `AuthGuard fullBleed`, mobile-first по паттерну `student-homework-problem-screen`.
2. **Лента занятий**: ближайшие/последние сверху. Карточка занятия:
   - Дата + время + тема.
   - Чипы материалов: «Запись» (→ Drive в новой вкладке), «Конспект» (→ PDF), «Домашка» (→ guided-homework problem screen).
   - Пустое состояние: «Материалов пока нет».
3. Без календарной сетки, без редактирования. Только чтение + переходы.

**Anti-leak:** ученик видит материалы (они student-facing по дизайну), но `tutor_lessons.notes` и любые tutor-only поля НЕ отдаются (см. §6, §7).

## 6. Эскиз модели данных (детали — в SPEC)

Зеркалируем паттерн материалов ДЗ, не изобретаем формат.

- **Таблица `tutor_lesson_materials`** (черновик):
  - `id`, `tutor_id`, `lesson_id` (FK `tutor_lessons`), `group_session_id` (nullable — для групповых),
  - `material_kind` ∈ `{'drive_link','pdf','homework_ref'}`,
  - `url` (для drive_link / pdf-ref в dual-format `storage://...`), `homework_assignment_id` (FK, для homework_ref),
  - `title`, `sort_order`, `created_at`, `created_by`.
- **Storage bucket** `lesson-materials` (mirror `homework-task-images`): путь `tutor/{auth.uid()}/{lesson_id}/{uuid}.pdf`. Attachment refs только через `parseAttachmentUrls`/`serializeAttachmentUrls` (dual-format).
- **Видимость = на уровне занятия/сессии:** ученик привязан к `tutor_lessons` (`student_id` / `tutor_student_id`) или к `group_session_id` → видит материалы этого занятия. Per-student матрицы нет.
- **RLS:** репетитор CRUD своих (по `tutor_id`); ученик SELECT материалов занятий, где он привязан. `tutor_lessons` уже даёт ученику SELECT своих занятий (RLS `student_id = auth.uid()`).
- **Edge function** для student-чтения (service_role, column-whitelist, никогда `notes`) — а не голый PostgREST (mirror `handleGetStudentProblem`, rule 40 / 97 error-contract).

> Точные имена колонок, миграции, GRANT-whitelist и RLS — финализируются в `spec.md`.

## 7. Контракт интеграции со `student-progress` (координация)

`student-progress` строится параллельно (spec/дизайн готовы, билд pending). Чтобы не было дрейфа и двойного учёта:

1. **Единый источник правды по ДЗ.** Занятие ссылается на тот же `homework_tutor_assignments`, что агрегирует `tutor-progress-api`. schedule-materials **не дублирует** баллы/статусы ДЗ — только хранит ссылку. Цепочка «занятие → ДЗ → балл» получается бесплатно.
2. **Оболочку вкладок на `/tutor/students/:id` владеет `student-progress`** (его spec §4.2 создаёт страницу с вкладками «Прогресс»/«Отчёт»). schedule-materials **добавляет сестринскую вкладку «Занятия»**, не создаёт вторую страницу. Если schedule-materials поедет раньше — он создаёт shell, progress добавляет свои вкладки (договориться о владельце shell на старте билда).
3. **Разные таблицы/потоки** (`tutor_lesson_materials` ≠ `tutor_student_targets` / task_states) → фичи не блокируют друг друга. Пересечение только в (а) tab-контейнере и (б) общих storage/attachment-паттернах.
4. **Без двойного write-path** в данные прогресса: schedule-materials пишет только в `tutor_lesson_materials`; ничего в `homework_tutor_*` state не трогает.

## 8. Success-метрики

**Leading (за 1-2 недели пилота):**
- % занятий с ≥1 прикреплённым материалом (цель: Вадим крепит к большинству прошедших уроков).
- Время крепления одного материала (цель: ≤ ~15 сек на drive-ссылку).
- % учеников, открывших «Среду» ≥1 раз; число переходов «Запись»/«Конспект».

**Lagging:**
- Миграция реального ученика Вадима на платформу (снят блок-фактор «некуда») `[L2214-2216]`.
- Снижение отправок материалов через ТГ (Вадим перестаёт дублировать в Telegram).
- Качественный сигнал родителя/ученика про «солидность пакета» (P3).

## 10. Фазинг

- **Phase 1 (v1, этот scope):** крепление материалов к занятию (drawer) + лента «Среда» у ученика + вкладка «Занятия» на странице ученика + уведомление.
