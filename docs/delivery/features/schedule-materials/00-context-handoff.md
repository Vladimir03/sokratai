# Часть 1 · Playbook: обработка Job репетитора (best practices)

Канон процесса — `docs/discovery/development-pipeline.md` (8 шагов). Ниже — дистилляция того, что сработало на фиче `student-progress`, с акцентами.

### Поток (Cowork → Claude Design → Claude Code)
1. **INTAKE → Job Ticket** (пайплайн шаги 1–2): идея в `backlog.md` → привязка к Job + Go/Wait/Kill + pilot-фильтр. Нет привязки к Job — Wait/Kill.
2. **Custdev extract** (deliverable `01-*`): дословные цитаты по каждому репетитору с `[L###]`-ссылками на транскрипт; срез по 3 сегментам (репетитор/ученик/родитель); синтез общего Job + расхождений; открытые вопросы. **Каждое продуктовое утверждение → конкретная цитата.**
3. **AJTBD-PRD (Замесин)** (`prd.md` / `02-*`): маппинг на Job Graph **кодами** (R/S/P + sub-jobs), 4 силы прогресса, сегменты, scope IN/OUT/LATER, success-метрики leading/lagging, open questions (blocking/non-blocking).
4. **Раунды вопросов ДО стройки** (`AskUserQuestion`): закрыть крупные развилки (семантика, IA, scope, доступ сегментов) — с рекомендацией в каждой. Не строить, пока форки открыты.
5. **Claude Design** (бандл `design-handoff/`): запинить дизайн-систему + репо + live-URL; **один экран за раз**, 3 layout-варианта → выбор → mobile+состояния; валидация на 3 персонах; **переиспользовать существующие компоненты**; честные шкалы / anti-leak / Safari зашиты в бриф.
6. **SPEC** (`spec.md`, шаблон `FEATURE-SPEC-TEMPLATE.md`): **Section 0 Job Context** (R/S/P + граф-ссылки), AC testable (Given/When/Then), приоритеты P0/P1, карта переиспользования, **anti-leak + Safari как инварианты**, Parking Lot, фазинг.
7. **TASKS** (`tasks.md`, шаг 5): `TASK-N` (Job / Agent / Files / AC) + обязательная секция **copy-paste промптов** по doc 19/20 (Role / Context / Canonical docs / Task / AC / Guardrails / Mandatory end block).
8. **BUILD** (шаг 6): Claude Code (автор, plan mode) → `lint && build && smoke-check` → **Codex ревью чистой сессией** (без контекста автора) → fix → merge → `deploy-sokratai` → пилот → **LEARN** (signal → doc → spec, не signal → code).

### 10 принципов, которые держат качество
1. **Voice-of-customer first**, всё `[L###]`-трассируемо.
2. **Job Graph коды** (R/S/P); фича, закрывающая 2–3 участников = сильный приоритет (B2B2C-overlap).
3. **3-сегментная линза** всегда, но **режь неподтверждённое** (мы вырезали «ученик-зеркало» — кастдев его не просил).
4. **Спрашивай до стройки**, давай рекомендацию в каждой развилке.
5. **Переиспользуй, не перерисовывай** — против дрейфа в зрелом репо (грепни компоненты перед новым кодом).
6. **Anti-leak / Safari / dual-write** — first-class инварианты, не «потом».
7. **Честная модель данных** — без фейковых метрик (поймали «средний /5»); present-day факты (таблицы ФИПИ) проверяй поиском и **версионируй** (`scale_year`).
8. **Фазинг** — наименьший срез, дающий пользу пилоту, первым.
9. **Канон путей и имён** — `docs/delivery/features/{feature}/`: `prd.md`, `spec.md`, `tasks.md` (в корне репо `C:\Users\kamch\sokratai\docs`, НЕ в подпапке `SokratAI\docs`).
10. **Producer = consumer хэндофф** — бандл Claude Design + `spec.md` → Claude Code; Codex судит «свежим взглядом» по спеке + `.claude/rules`.

# Часть 2 · Kickoff: фича «Материалы в расписании» (Вадим)

### Запрос (дословно, кастдев Вадима)
> «Материалы в расписании — прикрепить к занятию ссылку Google Drive (видеозапись) + PDF (конспект) + ДЗ. Ученик заходит в "Среду" → видит тему + все материалы. Блокирующий фактор: уже есть ученик для миграции, но "пока некуда".» `[L1543, L2214-2216]`

Подкрепляющее:
- «Мне нужно оптимизировать **хранение записей**. Записи всё на Google Drive, но надо структурно отсылать ученикам… ТГ — так себе.» `[L1541]`
- Killer-job для него — **НЕ проверка, а хранение материалов** `[L1603]`.
- «Ученик заходит в расписание → видит "Среда": тема урока + конспект (PDF) + ссылка на запись (Google Drive) + домашка.» `[L1609]`
- Бонус: при многих группах ученик видит **общее расписание** и может присоединиться к другому занятию по теме `[L1686]`.
- Контекст абонемента: «запись + ДЗ на платформе = защита от споров с родителями» `[L1563]`.
- Mindset: **Co-builder / перевёрнутый класс**, хочет Сократ основной платформой. RICE #8 (Impact 3, его личный P0, блок-фактор миграции).

### Job-маппинг (AJTBD)
| Участник | Core Job | Sub-job | Как закрываем |
|---|---|---|---|
| Репетитор | R4 — Контроль/качество при масштабировании (перевёрнутый класс) | R4-2 (доставка через единый канал), R4-3 (база материалов) | материалы к занятию в одном месте |
| Школьник | S3 — Удобство, «всё в одном» | S3-1 (получить в привычном месте), S3-2 (не переключаться) | «Среда» = тема + запись + конспект + ДЗ |
| Родитель | P3 — ROI / «солидность пакета» | P3-1 (между уроками есть система) | «у нас платформа с записями и конспектами» |

> **Важно (LEARN-сигнал):** материалы-в-расписании — это **LMS-обвязка**, она шире текущего ядра графа (doc 04 §9 относил «full LMS» к expansion). Кандидат на **новый Core Job в Графе** («R5 — LMS: хранение и доставка материалов урока»). В новом чате — зафиксировать как обновление Job Graph (шаг 8 LEARN), а не натягивать на существующие коды.

### Что УЖЕ есть (из ответов Владимира — фича цепляется к этому, не строится с нуля)
- ✅ **Расписание у репетитора** — `tutor_lessons` (есть `start_at`), `TutorSchedule.tsx` (⚠ high-risk файл, rule 10). Точка крепления материалов.
- ✅ **Материалы к ДЗ** — `homework_tutor_materials` (PDF / ссылки / картинки). **Готовый паттерн для зеркалирования.**
- ✅ **Хранилище файлов** — Supabase Storage bucket (для PDF-конспектов).
- ❌ **Ученик НЕ видит своё расписание / «Среду»** — этого нет. **Это и есть главный объём работы и effort-драйвер.**

### Гипотеза scope (уточнить в новом чате)
**IN (v1):**
- Репетитор прикрепляет к занятию: **ссылку Google Drive** (запись — просто URL, без OAuth), **PDF-конспект** (upload в Storage), **связь с существующим ДЗ** (`homework_tutor_assignments`).
- **Student-facing «Среда»** — ученик видит свои занятия (тема + дата + материалы + ДЗ). ← основной билд.

**OUT (v1):**
- Интеграция Google Drive API/OAuth (храним только ссылку, без превью/синка).
- Видеохостинг (только ссылка на Drive).
- Cross-group «присоединиться к чужому занятию» (бонус Вадима `[L1686]`) → **LATER**.
- Сохранение прогресса ДЗ при паузе (Cloud.Text-баг `[L1660]`) → отдельная фича.
- Billing/абонементы.

**LATER:** cross-group видимость/join; общий каталог записей по темам.

### Карта переиспользования (с учётом текущей разработки `student-progress`)
- `tutor_lessons` / `TutorSchedule.tsx` — точка крепления (аккуратно, high-risk).
- `homework_tutor_materials` — **зеркалировать** как `tutor_lesson_materials` (PDF/link/ДЗ-ref), не изобретать формат.
- Supabase Storage + helpers `parseAttachmentUrls`/`serializeAttachmentUrls` (dual-format) — для PDF.
- **Из `student-progress` (строится сейчас):**
  - **Q4-решение Владимира = переиспользовать surfaces:** материалы видны и в расписании, и на **странице ученика** `/tutor/students/:id` (новая вкладка «Занятия/Материалы» рядом с «Прогресс»/«Отчёт»).
  - Student-facing «Среда» строить по паттернам **student-homework-problem-screen** (mobile-first, `AuthGuard fullBleed`, `useIsMobile`, 100dvh, Safari rule 80) — там уже отработан ученический mobile-UX.
  - Anti-leak попроще: материалы **по дизайну student-facing** (в отличие от рубрик), но tutor-only заметки к занятию не светить.
- **Не блокирует и не блокируется** `student-progress`: разные таблицы/потоки → можно делать параллельно/после v1.0 прогресса. Пересечение — только UI на странице ученика (вкладка) + общие storage/attachment-паттерны.

### Открытые вопросы для нового чата (закрыть до SPEC)
| Вопрос | Кто | Блок? |
|---|---|---|
| «ДЗ к занятию» = ссылка на существующий `homework_tutor_assignment` или новая сущность? | продукт | да |
| «Среда» = полноценный экран расписания ученика или per-lesson страница занятия? | дизайн+продукт | да |
| Ученик видит только свои занятия или общее расписание групп (бонус) в v1? | продукт | да (scope) |
| Google Drive — только хранимый URL (ссылка-кнопка) или embed-превью? | инженерия | нет |
| Материалы на странице ученика — отдельная вкладка или секция в «Прогресс»? | дизайн | нет |
| Доступ: запись/конспект виден всем в группе или индивидуально? | продукт | да |

### Эффорт-флаг
Это **не** «прикрутить ссылку» — основной объём = **student-facing расписание/«Среда»**, которого сегодня нет. Заложить как L (>2 дней), вероятно разбить на фазы: Phase 1 (прикрепление материалов к занятию + минимальный student-просмотр), Phase 2 (полноценная «Среда» + cross-group LATER).
