## Section 0: Job Context

**Кто нанимает продукт:** Vladimir (фаундер/CEO) — единственный, кто принимает решения по wedge, ценам, сегменту и приоритетам (AGENTS.md → Roles).

**Core Job:** «Когда я решаю, куда потратить следующую неделю разработки и маркетинга, я хочу видеть, где именно рвётся путь репетитора от регистрации до оплаты и какой канал приводит тех, кто доходит до денег, — чтобы вкладываться в причину, а не в симптом».

**Ситуация (на момент запуска):** 3 платящих репетитора, ~50 зарегистрированных, ~150 учеников — pre-PMF. Главный необъяснённый факт: **почему из 50 репетиторов платят 3**.

**Почему не полный AARRR (решение владельца):** на 5–10 регистрациях в неделю проценты конверсии — статистический шум (два человека меняют «40% → 20%»). Решения принимаются не по агрегатам, а по **именам**: кто застрял, на каком шаге, из какого канала. Отсюда ядро = поимённая воронка активации + минимальная шапка здоровья, а не пять экранов метрик. Подход «do things that don't scale» — дашборд как рабочий список, а не отчёт.

**Definition of done для Job:** открыв `/admin`, фаундер за 30 секунд видит здоровье бизнеса и за один клик получает список конкретных людей, которым надо написать сегодня.

## 1. North Star и метрики

**NSM — «Weekly Value Tutors»**: репетиторы, чей ученик реально сдал ДЗ за последние 7 дней (продукт закрыл Job репетитора). Выбран как опережающий индикатор будущих оплат: репетитор платит не за фичи, а за то, что ученик решает.

**Шапка здоровья (6 цифр):** MRR (run-rate) · Платящие · Триалы · WAU репетиторов · Новые за 7д · NSM. У каждой — дельта к прошлой неделе и tooltip с формулой.

**Воронка активации (ядро):**

| # | Ступень | Тип | Что означает для фаундера |
|---|---|---|---|
| 1 | Регистрация | поведенческая | пришёл |
| 2 | Добавил ученика | поведенческая | увидел смысл заводить базу |
| 3 | Создал ДЗ | поведенческая | попробовал главный инструмент |
| 4 | Отправил ДЗ | поведенческая | доверил продукту отношения с учеником |
| 5 | Ученик открыл | поведенческая | ученик дошёл (проверка канала доставки) |
| 6 | Ученик сдал | поведенческая | **ценность состоялась** (= NSM-событие) |
| ₽ | Триал | коммерческая (независимая) | был триал |
| ₽ | Оплата | коммерческая (независимая) | заплатил хоть раз |

Клик по ступени → поимённый список (имя, канал, дата регистрации, последняя активность, статус, ссылка на Telegram). Для 1–6 показываются **застрявшие ровно здесь**, для «Триал» — «был триал, но не оплатил», для «Оплата» — дошедшие.

**Каналы привлечения:** канал → всего / дошли до ценности / оплатили (исторические факты). Каналы v1: «Егор (QR)» (`registration_source='egor'` ∨ `promo_code='BLINOV_20'`), «Реф: …», «Органика», «Без атрибуции».

**В зоне риска:** платящие/триальные без сдач учеников за 7 дней или с ручной меткой `at_risk` + ручные CEO-теги (`tutor_pilot_crm`: готовность платить, риск, key pain).

## 2. Ключевые решения (locked)

1. **Триал/оплата НЕ входят в поведенческую воронку.** `trial_ends_at` выдаётся при регистрации автоматически → иначе репетитор без единого ученика «проходил» бы все продуктовые ступени (P0 ревью ChatGPT-5.6).
2. **Каналы — только исторические факты**, не текущий статус (оплативший исчезал бы из «триалов», ручной грант выглядел бы конверсией канала).
3. **Источник воронки — доменные таблицы, не `analytics_events`** (та существует с 2026-07-01 → не покрывает существующие когорты).
4. **Ретеншн/воронка «Аналитики» — только само-зарегистрированные** (placeholder-ученики репетитора имеют `created_at` = дата заведения карточки).
5. **Все календарные сутки — московские** (аудитория и фаундер в РФ).
6. **Аудитория дашборда — Vladimir + Егор** (партнёру важен результат его офлайн-активностей).

## 3. Реализация

**Stage 1 — «Пульс» + чистка админки (СДЕЛАНО).**
- Backend: `supabase/functions/_shared/ceo-pulse.ts` (`computePulse` — единый агрегатор, переиспользуется Stage 2) + edge `admin-ceo-dashboard`.
- Frontend: `src/components/admin/pulse/*`, вкладка «Пульс» первой в `src/pages/Admin.tsx`.
- Чистка: удалены вкладки Бизнес / Открытия / ДЗ (дубль Репетиторов) / Пробники (осталась выжимка `AdminAiQuality`); ценное из «Бизнеса» (at-risk, CRM-теги) переехало в Пульс; Тарифы — платящие/триал наверх.
- Аудит «Аналитики» (`admin-analytics`): 6 исправленных расчётов + 5 фиксов ревью — детали в rule 101.

**Stage 1.5 — Пре-воронка «до регистрации» (СДЕЛАНО 2026-07-15, решения владельца).** Ступени: визит лендинга → клик CTA → открыл форму (+ QR-визиты `/egor` отдельной строкой); источник — **Яндекс.Метрика API** (данные уже собирались целями `tutor_landing_*`, включая историю); показ — блок в Пульсе над воронкой + недельный дайджест; без UTM-разбивки (на текущем трафике — шум). Агрегатно/анонимно by design. Реализация: `_shared/metrika.ts` + `PulsePreFunnel.tsx`; инварианты (цели Метрики, missingGoals, three-places-правило для нового CTA) — rule 101. Ops: секрет `METRIKA_API_TOKEN`.

**Stage 2 — Telegram-дайджест (СДЕЛАНО 2026-07-15).** Edge `ceo-telegram-digest` (guard `SCHEDULER_SECRET`, `verify_jwt=false`, реюз `computePulse`), `_shared/telegram-send.ts` (извлечён из `tutor-plan-expiry-reminder`), получатели — секрет `CEO_DIGEST_CHAT_IDS`, идемпотентность — `ceo_digest_log` (миграция `20260715160000`, claim-first), два cron через Lovable Management API (rule 95 — НЕ миграцией):
- **Недельный** (пн 07:00 МСК): шапка + пре-воронка + движение воронки за неделю + топ-3 «кому написать» (at-risk → застрявшие).
- **Дневной** (08:00 МСК): только при событиях (новая регистрация с каналом, оплата тарифа по моменту активации, новый триал). **Пусто → не слать** (тишина = ничего не произошло).
- Детальные инварианты — rule 101.

**Stage 3 — Рефералка v1, attribution-only (не начат).** `tutors.referral_code` (авто всем + backfill), `profiles.referred_by_code` (first-write-wins), `profiles.utm_source`. **Не переиспользовать `profiles.promo_code`** (занят BLINOV_20 first-write-wins + anti-leak: `promo_code` не возвращается клиенту, а реферальный код обязан быть видим). Захват: `?ref_code=` + поле на регистрации + позже в профиле. Money-путь `yookassa-*` НЕ трогаем — только маркер-комментарий.
**Будущее (отдельная спека, money-critical):** новичку −15% на первые 2 оплаты, рефереру 10% с его оплат до года.

## 4. Известные ограничения

- ~~Возвраты не вычитаются из MRR~~ — **закрыто 2026-07-15, смержено в main** (миграция `20260715130000`, 2 раунда независимого ревью ChatGPT-5.6 — вердикт «можно мержить и деплоить»): `refund.succeeded` резолвится через `object.payment_id` из YooKassa API, возвраты пишутся в append-only `payment_refunds`, MRR считает net **time-aware** (снапшот `now−7d` не видит сегодняшний возврат → недельная Δ показывает момент падения). Попутно закрыты: слепой `status` из body (сброс интро-цены поддельным `payment.canceled`) и P0 retry-семантики (`.maybeSingle()` — транзиентный сбой БД больше не подтверждает оплату как «подделку» без активации). Механика + retry-таблица — rule 99. Ops: подписка на `refund.succeeded` в кабинете YooKassa **проверена и включена** (Vladimir, 2026-07-15; заодно URL уведомлений переведён на `api.sokratai.ru`).
- **Полностью возвращённый платёж всё ещё считается «первым платежом»** в интро-цене 200₽ (`yookassa-create-payment` смотрит на `status='succeeded'`, а статус при возврате не меняется). Осознанное решение: менять = продуктовый выбор владельца.
- `admin_list_tutor_plans.active_students` не фильтрует `archived_at` → может расходиться с ценовой вилкой тарифа (пре-существующее).
- Чат репетитор↔ученик (rule 100) пока не входит в сигнал активности «Аналитики».
