## 0. Заземление: что Сократ реально хранит по каждой проверке [✓ РЕАЛЬНО]

**Модель проверки:** Gemini через Lovable AI Gateway (`ai.gateway.lovable.dev`; в коде модель
`google/gemini-3-flash-preview`, temperature 0.2, timeout 35 с) — ⚠ версию модели сверить.

**Гранулярность:** одна AI-проверка = одна строка `homework_tutor_task_states` =
**(ученик × задача)** в рамках одного ДЗ (UNIQUE `(thread_id, task_id)`). Полный диалог проверки
(сдача + разбор) — в `homework_tutor_thread_messages`.

### Хранится и запрашиваемо из БД

| Что | Где | Примечание |
|---|---|---|
| `ai_score` (numeric, шаг 0.5) | `task_states` | балл, который поставил Gemini |
| `ai_score_comment` | `task_states` | обоснование AI — **tutor-only**, ученику не грантится |
| `ai_criteria_json` `[{label,score,max,comment,kind}]` | `task_states` | покритериальная разбивка, **только языки** (DELF/ЕГЭ EN/IELTS); физика/математика = `null` |
| `last_ai_feedback` | `task_states` | последний текст фидбэка |
| `earned_score` / `available_score` | `task_states` | итог с деградацией за подсказки/ошибки |
| `wrong_answer_count` / `hint_count` / `attempts` | `task_states` | счётчики |
| `submission_payload {numeric, photos[], text, voice_ref?}` | `thread_messages` (`message_kind='submission'`) | что сдал ученик |
| полный текст разбора AI | `thread_messages` (`message_kind` ∈ `check_result`/`ai_reply`) | сообщение-вердикт |
| фото решения ученика | bucket `homework-submissions`, ссылка `storage://…` в payload | файл, **не** текст |

### НЕ хранится как колонка — только в логах телеметрии (эфемерно)

| Сигнал | Статус |
|---|---|
| `verdict` (CORRECT / ON_TRACK / INCORRECT / CHECK_FAILED) | ❌ не колонка. Частично выводится из `status` (completed ≈ CORRECT) + `ai_score`, но **ON_TRACK vs INCORRECT из БД не отличить** |
| `confidence` (0–1) | ❌ только в событии `guided_check_success` |
| `error_type` (calculation/concept/factual/…) | ❌ в результате модели, не колонка |
| `failure_reason` (timeout / invalid_json / task_image_missing / …) | ❌ только логи (`guided_check_error`) |
| **OCR / распознанный текст рукописи** | ❌ **НЕ существует** (legacy `recognized_text` дропнут). Фото → Gemini мультимодально в реальном времени, текст не сохраняется |

> С 2026-06-30 «слой вердикта» (verdict/confidence/error_type/failure_reason) пишется ВПЕРЁД в новую
> таблицу `hw_ai_check_events` — см. `telemetry-hw-ai-check-events.md`. Ретроспективы по нему нет.

**Главный вывод для проекта:** категориальный вердикт, уверенность, тип ошибки и причина сбоя
исторически жили **только в логах**, а не в запрашиваемых колонках. Скоркарту качества нельзя
посчитать простым `SELECT` — нужна либо телеметрия (раздел 3), либо доступ к логам.

**Самый ценный готовый сигнал ошибки:** правка репетитора поверх AI —
`tutor_score_override ≠ ai_score`, плюс `tutor_force_completed_at`, `tutor_reviewed_at`. Это
единственный персистентный «человеческий лейбл правды».

**Период:** миграции Oct 2025 → Jun 2026 (~9 мес). Объёмы строк — ⚠ нужен доступ к БД. Пилот:
физика ЕГЭ — основной сегмент (полная рубрика ФИПИ), французский/DELF — покритериальная разбивка.

## 1. Каркас скоркарты качества одной проверки

**Единица оценки:** одна AI-проверка (строка `task_states` + сообщение `check_result`).

**Стратификация выборки** [⚠ доли подтвердить]: по `subject` (физика / язык / прочее),
`check_format` (`short_answer` / `detailed_solution`), `task_kind` (`numeric` / `extended` /
`proof` / `speaking`), наличию фото, языку ответа.

**Шкала: 0 / 1 / 2 по каждому измерению** (0 — неприемлемо, 1 — частично, 2 — хорошо) +
**бинарный «критический дефект»**, который перекрывает всё.

| # | Измерение | 0 | 1 | 2 | Источник истины |
|---|---|---|---|---|---|
| D1 | **Верность вердикта** (CORRECT/ON_TRACK/INCORRECT соответствует реальности) | вердикт неверен | спорно/граница | верно | эталон `solution_text` + правка репетитора (`tutor_score_override`) |
| D2 | **Корректность пошагового разбора** (если `detailed_solution`) | разбор ошибочный/выдуман | неполный | корректный и полный | ручная аннотация эксперта |
| D3 | **Качество комментария** (конкретность, польза, без утечки решения) | пусто/вода/утечка | общий | конкретный, по делу | аннотация + флаг `detectLeak` |
| D4 | **Точность распознавания** (AI верно «прочитал» ответ/фото) | прочитал неверно | частично | верно | ручная сверка фото ↔ что AI «увидел» |
| D5 | **Тон/сложность для возраста (16–18)** | груб / слишком сложно | нейтрально | поддерживающе и по уровню | аннотация |
| **КД** | **Критический дефект** = ложно-«верно» на неверном решении (D1 fail при высоком `ai_score`) | — | — | — | override вниз / эксперт |

**Агрегат на выборке:** доля проверок без критического дефекта (главная safety-метрика) → средний
D1 (точность вердикта) → D2–D5 как качество объяснения. **Калибровка** (опционально): соответствие
`confidence`/`ai_score` реальной верности (reliability-кривая) — `confidence` берётся из телеметрии.

**Как считать:** выборка N проверок, **2 аннотатора + reference** (эталон `solution_text`, где есть;
правка репетитора как «серебряная истина»), межаннотаторское согласие (Cohen's κ), отчёт error-rate
по типам с доверительными интервалами. ⚠ На пилотном объёме N мал → планировать ручную аннотацию
(десятки–сотни кейсов), а не «большую статистику».

### Слой «ценность» (не только качество)

Качество → доверие → внедрение. Метрики ценности из реальных полей/событий:

| Метрика | Как считать | Поле/событие |
|---|---|---|
| **Auto-accept rate** | доля проверок, где репетитор не правил (`tutor_score_override IS NULL` среди reviewed) | `task_states` ✓ |
| **Override rate / направление** | доля и направление правок (вниз = ложно-верно, вверх = ложно-ошибка) | `tutor_score_override − ai_score` ✓ |
| **Доля «не смог проверить»** | CHECK_FAILED среди проверок | `hw_ai_check_events` / `guided_check_error` |
| **Латентность проверки** | время ответа модели | телеметрия / ✚ логировать `latency_ms` явно |
| **Доезд до ученика** | открыл ли ученик результат, пересдал ли | `student_opened_at`, повторные `submission` ✓ |

## 2. Таксономия failure modes чекера (заземлённая на Сократ)

| Тип | Определение | Как детектировать (реальный сигнал) | Гипотеза влияния на доверие |
|---|---|---|---|
| **F1. Ложно-«верно» (false accept)** | AI зачёл неверное решение | `ai_score` высокий **И** `tutor_score_override < ai_score` (правка вниз) / force-complete с правкой | **Критично.** Ученик закрепляет ошибку → доверие репетитора падает мгновенно |
| **F2. Ложно-«ошибка» (false reject)** | AI забраковал верное | `tutor_score_override > ai_score` (правка вверх) | Демотивирует ученика; репетитор обязан вмешиваться |
| **F3. Пропуск ошибки в развёрнутом** | `detailed_solution`: AI CORRECT, решение неполное | `check_format='detailed_solution'` + частичный override | Подрывает ценность «развёрнутой» проверки |
| **F4. Галлюцинация комментария** | AI ссылается на несуществующее («только фото условия» при наличии решения; выдуманные шаги) | `guided_check_task_image_missing` (известный класс) + ручная аннотация | Сильный сигнал «AI не смотрел работу» |
| **F5. Сбой распознавания рукописи** | AI неверно прочитал фото на бумаге | **Нет авто-сигнала** (OCR не хранится) → ручная сверка + косвенно `image_fetch_failed`, повторные сдачи того же фото | Высокий драйвер недоверия для сегмента «решаю на бумаге» |
| **F6. Неверный пошаговый разбор** | логически неверная цепочка при верном итоге | Ручная аннотация (D2) | Опасно: учит неправильному методу |
| **F7. Утечка решения** | фидбэк/подсказка выдаёт ответ | `check_solution_leak_rejected`, `hint_solution_leak_rejected` (частота из логов ✓) | Ломает педагогику guided-режима |
| **F8. Бесполезная подсказка** | вода / запрещённые фразы | `hint_rejected`, `hint_fallback_used` (логи ✓) | Раздражает, снижает вовлечённость |
| **F9. Неуместный тон/сложность** | грубо или не по возрасту/уровню | Ручная аннотация (D5) | Тихий отток ученика |
| **F10. Инфраструктурный сбой** | модель не ответила | `guided_check_error{failure_reason}` / `hw_ai_check_events` | «Не смог проверить» = видимый сбой |

**Опора на правки репетитора (F1/F2/F3)** — самый ценный готовый источник лейблов: персистентен и
запрашиваем. ⚠ Работает только если репетиторы реально правят — если почти не правят, лейблов мало
(открытый вопрос 6.3).

**Защиты, которые уже в коде** (значит частоту F4/F7/F8/F10 можно мерить из логов сразу):
утечка решения → `detectLeak` (verbatim для гуманитарных, token-overlap для STEM) → retry → scrub;
бесполезная подсказка → `FORBIDDEN_HINT_PHRASES` → validate → retry → fallback; фото не загрузилось →
fail-closed `CHECK_FAILED(task_image_missing)`. ⚠ Низкая `confidence` **не влияет на вердикт** (порога
отклонения нет) → «уверенно неверно» сейчас не отлавливается.

## 3. Спека событий для мониторинга качества в проде

**Реализовано 2026-06-30 — таблица `hw_ai_check_events`** (см. `telemetry-hw-ai-check-events.md`):
два события — `check_completed` (verdict/confidence/error_type/failure_reason/ai_score + контекст) и
`tutor_correction` (override_delta + тип правки). Пишется из `homework-api` под service_role,
PII-free, копит вперёд. **Это и есть «слой B», которого нет ретроспективно.**

Дополнительно уже эмитятся (server `console.warn`, PII-free): `guided_check_task_image_missing`,
`check_solution_leak_rejected`, `check_criteria_comment_leak_scrubbed`, `guided_check_fast_path_match`,
серия `hint_*`. Клиентские (реестр `src/lib/homeworkTelemetry.ts`): `guided_answer_correct/incorrect/
check_failed`, `student_submission_verdict`, `manual_score_override_saved`, `task_reviewed`.

**Дерево метрик:** error-rate по типам (F1–F10), auto-accept rate, направление override,
CHECK_FAILED-rate, leak/fallback-rate, латентность, время-до-проверки.
