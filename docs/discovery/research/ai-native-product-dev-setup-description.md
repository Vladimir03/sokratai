# AI-Native Product Development Setup

> Описание системы для визуализации и презентации в AI-комьюнити.
> Автор: Владимир Камчаткин, CEO SokratAI
> Дата: 2026-03-26

---

## Что это такое (в одном абзаце)

Полностью AI-управляемый цикл разработки продукта — от исследования рынка до работающего кода — в котором AI-агенты выполняют 90% операционной работы, а фаундер остаётся стратегическим decision-maker. Не «используем ChatGPT для брейнштормов», а **замкнутая система**, где результат одного AI-этапа становится входом для следующего, а код пишется агентами, которые читают те же продуктовые документы, что и фаундер.

---

## Архитектура системы: 4 слоя

### Слой 1 — Методологическая база (Knowledge Layer)

**Что:** база знаний по продуктовой методологии AJTBD (Advanced Jobs To Be Done) Ивана Замесина + мета-фреймворк AURA.

**Как устроена:**
- 3 методологических документа — концепции AJTBD, фреймворк AURA, готовые промпты
- Конспекты вебинаров оцифрованы в структурированные markdown-документы
- Каждый документ — не заметка, а **операционный ресурс**: с ним можно сразу работать, давать AI, строить на его основе промпты

**Файлы:**
- `01-ajtbd-core-concepts-zamesin.md` — ядро методологии
- `02-aura-meta-framework-zamesin.md` — AURA (AJTBD + Unit Economics + RAT + ABCD)
- `03-ajtbd-prompts-zamesin.md` — 6 готовых промптов от автора методологии

**Зачем:** AI-агенты не могут изобрести методологию. Но если дать им правильно оцифрованную методологию — они выполнят по ней исследование быстрее и системнее, чем человек.

---

### Слой 2 — AI-исследование рынка (Research Layer)

**Что:** цепочка AI-промптов, каждый из которых генерирует структурированный исследовательский артефакт. Промпты содержат полный методологический контекст (из Слоя 1), так что AI не фантазирует, а работает по фреймворку.

**Конвейер (Pipeline):**

```
Промпт B2C-сегментация          Промпт B2B-сегментация
(родители + ученики)              (репетиторы как микро-бизнес)
         │                                  │
         ▼                                  ▼
   5 B2C-сегментов                   5 B2B-сегментов
   с Core Jobs, Big Jobs,            с Core Jobs, TAM/SOM,
   TAM/SOM, критериями               барьерами переключения
         │                                  │
         └──────────┬───────────────────────┘
                    ▼
         Промпт кросс-анализ B2C×B2B
         (матрица 5×5, скоринг по 3 осям)
                    │
                    ▼
            TOP пересечения B2B2C
            (сегмент × сегмент × overlap работ)
                    │
                    ▼
            Граф работ (Job Graph)
            для каждого TOP-пересечения
            (все участники, все sub-jobs,
             точки overlap, КПР)
```

**Ключевой принцип:** каждый следующий промпт **встраивает результаты предыдущего**. Кросс-анализ содержит полные данные обеих сегментаций. Граф работ содержит данные кросс-анализа. Нет потери контекста между шагами.

**Инструменты:**
- ChatGPT Deep Research — для сегментаций (доступ к свежим данным рынка, статистика)
- Claude Opus — для кросс-анализа и графов работ (лучше в структурном мышлении, длинный контекст)

**Артефакты (уже созданы для SokratAI):**
- B2C-сегментация: 10 сегментов с полным AJTBD-анализом
- B2B-сегментация: 5 сегментов с Core Jobs и барьерами
- Кросс-анализ: матрица пересечений с 3-мерным скорингом
- Граф работ B2B2C: 37 подработ для 3 участников, 10 Core Jobs, точки overlap

### Слой 3 — Продуктовые документы (Product Layer)

**Что:** 20 операционных документов, покрывающих весь путь от «зачем мы это делаем» до «какой отступ у кнопки». Выстроены в каноническом порядке чтения — и для человека, и для AI-агента.

**Каноничный стек (canonical read order):**

| # | Документ | Что решает |
|---|----------|-----------|
| 01–07 | AJTBD Research | Кто клиент, какие работы, текущие решения, value hypotheses |
| 08 | Wedge Decision Memo | **Входная точка продукта** — одна конкретная работа, которую мы делаем 10× лучше |
| 09 | WTP/Pricing Memo | Готовность платить, ценовые якоря |
| 10 | Concierge Pilot Plan | План проверки гипотез на живых пользователях |
| 11 | Value Prop One-Pager | Позиционирование в одном предложении |
| 12 | Landing / Commercial Draft | Коммерческое предложение |
| 13 | PR/FAQ One-Pager | Amazon-стиль описание продукта из будущего |
| 14 | **PRD** | Product Requirements — что строим, scope, non-goals |
| 15 | **JTBD Scenarios Backlog** | Сценарии по работам — декомпозиция PRD |
| 16 | **UX Principles** | Принципы интерфейса (38 KB правил) |
| 17 | **UI Patterns & Component Rules** | Конкретные UI-паттерны и компоненты (54 KB) |
| 18 | Pilot Execution Playbook | Операционный план пилота |
| 19 | **Agent Workflow & Review System** | Как AI-агенты работают в этой системе |
| 20 | **Claude Code Prompt Patterns** | Шаблоны промптов для запуска разработки |

**Критическая идея:** документы 14–17 — это **канон**, который AI-агент обязан прочитать перед написанием кода. Не рекомендация, а жёсткое правило в `CLAUDE.md`. Агент, который пишет UI без документа 17, создаст «типовой чат-UI» вместо того, что нужно продукту.

### Слой 4 — AI-разработка кода (Engineering Layer)

**Что:** AI-агенты (Claude Code, Codex, Lovable) пишут код, но внутри жёсткой системы ограничений и правил, которые привязывают каждую строку кода к продуктовым документам.

**Ключевые механизмы:**

**a) CLAUDE.md — «конституция» для AI-агента**
- Лежит в корне репозитория
- 500+ строк правил: архитектура, безопасность, запрещённые паттерны, critical files
- AI-агент читает его **при каждом запуске** и обязан следовать
- Содержит: Safari-совместимость, React Query conventions, performance rules, DB schema constraints

**b) Canonical docs chain**
- Перед любой tutor-фичей агент читает 7 документов в фиксированном порядке
- Не прочитал PRD и UX Principles → не начинаешь писать код
- Это гарантирует, что код усиливает wedge, а не расплывается

**c) Anti-drift guardrails (анти-дрифт)**
- Запрет: превращать `Помощник` в generic chat
- Запрет: добавлять AI-output без action layer
- Запрет: добавлять фичи без Job-based justification
- Запрет: менять архитектурные решения (Student/Tutor isolation)

**d) Review system**
- Первый агент реализует
- Второй агент (или тот же в новой сессии) ревьюит против canonical docs
- Человек (фаундер) принимает merge decision

**e) Feature specs**
- Каждая фича имеет свой spec в `docs/features/specs/`
- Spec содержит: JTBD-обоснование, фазы, acceptance criteria
- Агент реализует фазу → human review → следующая фаза

## Что делает эту систему «AI-native» (а не «AI-assisted»)

### AI-assisted (как делают большинство)
```
Человек думает → Человек пишет промпт → AI генерирует текст → Человек копирует в Google Doc → Человек думает дальше
```
AI — инструмент. Между AI-шагами — ручная работа и потеря контекста.

### AI-native (что мы построили)
```
Методология → [AI] Сегментация B2C → [AI] Сегментация B2B
    → [AI] Кросс-анализ (input: оба результата)
    → [AI] Граф работ (input: кросс-анализ + методология)
    → Продуктовые документы (PRD, UX, UI patterns)
    → [AI Agent] Feature spec
    → [AI Agent] Код (читает PRD + UX + UI patterns перед каждым коммитом)
    → [AI Agent] Review (читает те же docs, проверяет alignment)
    → Работающий продукт
```

**Разница:**
1. **Результат AI — вход для следующего AI.** Нет ручного copy-paste между этапами
2. **AI живёт в репозитории рядом с кодом.** Продуктовые docs — не в Google Docs, а в `docs/` рядом с `src/`. AI-агент читает их через файловую систему
3. **Методология закодирована в промпты.** AI не «брейнштормит» — он выполняет фреймворк AJTBD по шагам, потому что промпт содержит 10–20K символов методологического контекста
4. **Guardrails — в коде, не в голове.** `CLAUDE.md` физически не даст агенту сломать архитектуру или забыть Safari-совместимость. Это не «помни про Safari» — это «build.target: safari15 + список запрещённых API»
5. **Один человек = полноценная команда.** Фаундер-соло выполняет роли: product researcher + product manager + UX designer + frontend engineer + QA. AI-агенты выполняют 90% operational работы в каждой роли

## Визуальная метафора для схемы

**Воронка знания → действия:**

```
┌─────────────────────────────────────────────────┐
│            МЕТОДОЛОГИЯ (AJTBD + AURA)           │  ← Фреймворк
│          Вебинары → оцифрованные docs           │     (Knowledge)
└──────────────────────┬──────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────┐
│           AI-ИССЛЕДОВАНИЕ РЫНКА                 │  ← Discovery
│                                                 │     (Research)
│   [B2C промпт] ──→ B2C сегменты               │
│   [B2B промпт] ──→ B2B сегменты               │
│        └───┬───┘                                │
│            ▼                                    │
│   [Кросс-анализ] ──→ TOP пересечения           │
│            │                                    │
│            ▼                                    │
│   [Граф работ] ──→ Sub-jobs + overlap           │
└──────────────────────┬──────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────┐
│          ПРОДУКТОВЫЕ ДОКУМЕНТЫ                  │  ← Strategy
│                                                 │     (Product)
│   Wedge Memo → PRD → JTBD Scenarios            │
│   → UX Principles → UI Patterns                │
│   → Agent Workflow → Prompt Patterns            │
└──────────────────────┬──────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────┐
│         AI-РАЗРАБОТКА КОДА                      │  ← Delivery
│                                                 │     (Engineering)
│   CLAUDE.md (конституция) ────────────┐        │
│   Feature spec ───┐                   │        │
│                   ▼                   ▼        │
│            AI Agent #1 ──→ код                  │
│            AI Agent #2 ──→ review               │
│            Human ──→ merge decision             │
│                   │                             │
│                   ▼                             │
│           Работающий продукт                    │
└──────────────────────┬──────────────────────────┘
                       │
                       ▼
              Feedback от пилота
              → обратно в Research
```

**Ключевой визуальный акцент:** стрелки не только вниз — есть обратная петля от пилота в research. Это не водопад, а замкнутый цикл.

## Три инсайта для аудитории

1. **AI-native ≠ «пользуюсь ChatGPT».** Это архитектура, где AI — не инструмент, а операционная система. Результат одного AI-этапа программно становится входом следующего.

2. **Методология первична, AI вторичен.** Без AJTBD-фреймворка AI генерирует красивый, но бесполезный bullshit. С фреймворком — системное исследование, которое человек делал бы месяцами.

3. **Один человек может создать продукт уровня команды из 5–10 человек.** Но только если он построил правильную систему ограничений для AI-агентов. Парадокс: чем больше ограничений (CLAUDE.md, canonical docs, guardrails), тем лучше результат.

## Для кого это применимо

- Solo founders и micro-teams (1–3 человека)
- B2B и B2B2C продукты с глубоким customer research
- Продукты на стадии 0→1 (поиск PMF)
- Технические фаундеры, которые хотят product-led approach без найма продакта
- Нетехнические фаундеры, которые хотят строить с AI-агентами без найма разработчиков
