## Anti-leak — STATE-AWARE (do NOT copy homework's tutor-only model)

Mock-exam reveal depends on `attempt.status` — it is **not** "never show the student":

- **pre-submit** (`in_progress` / `paused`) → result endpoint refuses (409 / 410). `correct_answer` must never load into process memory.
- **post-submit, pre-approval** (`submitted` / `ai_checking` / `awaiting_review`) → reveal Часть 1 only (`correct_answer`, `kim_number`); Часть 2 = totals only.
- **post-approval** (`approved`) → reveal Часть 2 (`solution_text`, `tutor_score`, `tutor_comment`) — this IS the value proposition.
- **`ai_draft_json` / `ai_part1_ocr_json`** → never to the student, any status (tutor-only artifacts).

Contrast with homework: `solution_text` / `rubric_*` there are tutor-only **forever** (rule 40). If a reviewer flags "solution_text leak on the result page", confirm via the `status === 'approved'` gate — post-approval reveal is **by design**.

Endpoint hardening (`mock-exam-public`, `mock-exam-student-api`):
- Column-whitelisted SELECT (never `*`). Tutor-card whitelist = `name, avatar_url, bio, subjects` only — never `telegram_id`, `booking_link`, `email`.
- Conditional SELECT by `isApproved`: pre-approval must not load `task_text` / `solution_text` / `topic` into memory (defense-in-depth).
- Any new pre-submit status MUST be added to the result-endpoint early-reject and excluded from the post-submit allowlist (the `paused` leak, H1, came from missing this).
- Student RLS column guards: students may only write `earned_score IS NULL` + `score_source='student_form'` (blocks `score_source` spoofing).
- **✅ FIXED (2026-06-07, migration `20260607130000_mock_exam_variant_tasks_tutor_only_rls.sql`): `mock_exam_variant_tasks` SELECT is now TUTOR-ONLY.** Previously RLS was `FOR SELECT TO authenticated USING (true)` (migration `20260508120000`), so a logged-in **student** could `supabase.from('mock_exam_variant_tasks').select('correct_answer, solution_text, solution_image_urls')` from the browser console and read Part 1 answers + Part 2 reference solutions **mid-exam** → exam-integrity hole (pre-existing since the v1 schema; the column-whitelisting in the student edge functions is bypassed by direct PostgREST). The fix DROPs that policy and replaces it with `USING (public.is_tutor(auth.uid()))` — reusing the **existing** canonical SECURITY DEFINER helper `is_tutor` (migration `20260117211049`, = `has_role(uid,'tutor')`, already the gate for KB tutor-guard RLS and the same role `TutorGuard` checks; SECURITY DEFINER avoids the tutors-table-RLS subquery false-negative, rule 98). Zero frontend change: the only legit client reader is the tutor-only `MockExamVariantPreviewSheet` (under `TutorGuard` → passes `is_tutor`); the student taking-surface + result page use edge functions on **service_role**, which bypasses RLS. Students now get zero rows from a direct PostgREST read. **Invariant going forward:** any new client-side `mock_exam_variant_tasks` read must be tutor-only (the policy enforces it); student-facing data flows through the service_role edge functions, never direct PostgREST.

## AI graders — frozen contract, never auto-publish

`mock-exam-grade` runs as a background, fire-and-forget job from submit. Tutor approval stays mandatory; AI never AUTO-publishes the FINAL grade — but a **preliminary** AI result IS shown to the student post-submit (2026-06-02, item 2), clearly labelled, and the tutor still confirms/corrects.

- **Frozen JSON shape** (mirror in `src/types/mockExam.ts::MockExamPart2Draft`): `{ suggested_score: number|null, confidence: 'low'|'medium'|'high', elements_check:{I,II,III,IV}, comment_for_tutor, feedback, flags[], assigned_photo_indices?[] }`. Extend additively only.
- **`feedback` (2026-06-02, item 2)** — detailed «что верно/неверно» разбор shown to BOTH student and tutor (friendly, anti-spoiler). It is the ONLY `ai_draft_json` field besides `suggested_score` that may reach the student; `comment_for_tutor`/`flags`/`elements_check` stay tutor-only. Default `""` for pre-2026-06-02 attempts (re-grade to backfill).
- **Pre-approval Part 2 reveal (2026-06-02, item 2)** — `handleGetResult` now exposes, per task, `suggested_score` + `feedback` (extracted from `ai_draft_json`) + `solution_text`/`solution_image_urls` + `task_text`/`task_image_url` + `max_score`, labelled «предварительно — репетитор подтвердит». Reference solution revealed post-submit is intentional (one-shot exam, no retakes). `tutor_score`/`tutor_comment` still post-approval. NEVER expose `comment_for_tutor`/`flags`/`elements_check` to the student.
- Endpoint responses carry counters only for the **tutor list** — never raw `ai_draft_json`.
- **`mock_exam_variant_tasks.solution_image_urls` (2026-06-05, item 5)** — фото эталонного решения Часть 2, dual-format TEXT (single ref OR JSON-array, read via `parseAttachmentUrls`). Revealed to BOTH tutor and student post-submit (item 2) — INTENTIONAL for mock exams (one-shot exam; the эталон is the value-prop). Contrast homework, where `solution_image_urls` is tutor-only forever (rule 40). Content uploaded manually to bucket `mock-exam-variant-tasks/variant{N}/` + a resync migration per KIM. **Tutor PRE-assign preview (2026-06-07):** `MockExamVariantPreviewSheet` («Посмотреть условия задач» в `TutorMockExamCreate`) SELECTs `solution_image_urls` + resolves signed URLs **client-side** (`supabase.storage.createSignedUrl`, RU-safe via hardcoded `api.sokratai.ru`; mirror its `task_image_url` resolver) and renders a gallery in the Part 2 «Эталонное решение» collapsible — so the tutor sees reference-solution images BEFORE assigning. Tutor-only (RLS allows authenticated read of variant tasks incl. solutions).
- **State machine:** `in_progress → submitted → ai_checking → awaiting_review → approved`. Submit leaves status `submitted`; the grader CAS-claims `submitted → ai_checking → awaiting_review`. Never set `ai_checking` directly in the submit handler.
- **Stale-lock = 120s** in all 3 callsites (grader CAS, `retry-part1-ocr`, `regrade-part2`). `retry` / `regrade` reject a fresh `ai_checking` (409 GRADING_IN_PROGRESS) unless stale.
- **Bulk Часть 2** = two-pass: Pass 1 assigns photos→KIM (`buildBulkAssignmentPrompt`), Pass 2 grades per-KIM in parallel. Persisted tutor `assigned_photo_indices` are NOT overwritten on regrade.
- **Tutor status preservation:** the bulk path never overwrites rows `status IN (tutor_approved, tutor_modified)` — only `ai_draft_json`.
- **Approve never blocks:** missing scores auto-zero with a transparent `tutor_comment` (no `INCOMPLETE_*` 400s).
- **`total_score` is STORED, not derived** — heatmap (`MockExamHeatmap.tsx`) and student result API read the column directly, no recompute. **Invariant: when non-null, `total_score = COALESCE(total_part1_score,0) + COALESCE(total_part2_score,0)`.** Every per-task score write (`handleApproveTask` / `handlePart1ManualScore` / `handlePart1Finalize` / `handleRecheckPart1`) MUST call the **atomic RPC `mock_exam_resync_attempt_totals(_attempt_id)`** (migration `20260605140000`) right after the child upsert — via the `resyncAttemptTotals()` helper. The RPC recomputes `total_part1_score` (always) + `total_part2_score`/`total_score` (only when `total_score IS NOT NULL`, i.e. finalized) from the child tables in **one** UPDATE → last-writer-correct under concurrent post-approval edits (the prior JS read-sum-update was racy — ChatGPT-5.5 review P1). `handleApproveAll` still writes all three together at first approval. Do NOT reintroduce inline JS read-sum-update resync — always go through the RPC. Drift bug 2026-06-01 (recheck on `approved` left `total_score` stale → ИТОГО 19 not 21): backfill migration `20260601140000`. `manually_entered` is exempt (its `total_score` is tutor-entered, parts NULL; per-task handlers 409 it before the RPC).
- **Edit-after-approval (item 4, 2026-06-02)** — tutor can correct scores AFTER approval: `isReadOnly`/`isReadOnlyAttempt`/`isAlreadyapproved` score-edit gates are `manually_entered`-only; backend `handleApproveTask`/`handlePart1ManualScore`/`handlePart1Finalize` dropped the `approved` 409 (kept on OCR-retry + approve-all). Any post-approval edit MUST resync totals via the RPC above. The student result reads live scores → updates appear immediately.

## Part 1 — deterministic checker + OCR

- **Deno-mirror invariant (CRITICAL):** `src/lib/mockExamPart1Checker.ts` (browser preview + frontend) ↔ `supabase/functions/_shared/mock-exam-part1-checker.ts` (server submit / OCR / recheck) must stay logically identical. Grep both for `gradeMultiChoice` / `gradeOrdered` / `numericRoundingMatch` before merge. Symptom of drift: "preview says correct, final gives 0".
- **Partial credit (ФИПИ 2026):** `gradeMultiChoice` (KIM 5/9/14/18, set-error count) + `gradeOrdered` (KIM 6/10/15/17, Hamming distance) give 1 of max for a single error. All other modes binary — never bleed partial credit into strict / unordered / task20 / pair.
- **Strict-only rounding tolerance:** `numericRoundingMatch` is a fallback in `check_mode='strict'` only; scale = decimals in `correct`; never widen `student` beyond `correct`'s scale.
- **`score_source` enum** (`ocr` / `tutor` / `finalize_default` / `student_form`): every write to `mock_exam_attempt_part1_answers` must set it explicitly. The OCR re-run skip filter is `score_source === 'tutor'` only (never `earned_score IS NOT NULL`).
- **`ai_part1_ocr_json` canonical shape** = `{ cells:{[kim]:{value,confidence}}, __meta:{status:'success'|'failed', …} }`. Read via `.cells[kim]` + `.__meta.status`. Legacy rows pass through `normalizePart1OCRJson` in `handleGetAttempt`.
- **Tutor answer display resolves from BOTH sources (2026-06-02 fix):** the Часть 1 review must show the student answer as `student_answer ?? ai_part1_ocr_json.cells[kim].value` — typed/auto-saved first, else OCR-recognized. `TutorMockExamReview.tsx::resolvePart1StudentAnswer` is the single helper (used by `Part1SummaryCard` table, the counters, and both `Part1TaskDrillDownDialog` call-sites). Symptom of regressing this: tutor sees «без ответа» for a blank/OCR or NULL-`answer_method` attempt despite a real score (the form-card branch reads only `student_answer`). Counters: «верно/частично/неверно» from `earned_score` (`wrong` = `earned_score === 0` STRICTLY — never `?? 0`, else null-earned rows with a resolved answer mis-bucket as «неверно»), «не проверено» = `earned_score === null` + resolved present, «без ответа» = resolved value null. Never gate `wrong` on `student_answer !== null` (drops OCR rows → all counts read 0).
- **Blank thumbnail gating (2026-06-02):** `Part1SummaryCard` shows the uploaded ФИПИ blank / fallback photo ONLY when `answer_method !== 'form'` (i.e. null/legacy — where it may be a real OCR attempt mis-routed to the form card). Explicit `'form'` attempts may carry a leftover blank from a mode-switch — hide it (not grading-relevant; tutor-only, not a leak, but confusing).
- **Per-KIM score derive-on-read fallback (2026-06-06, extended to FORM mode 2026-06-07):** both the student result (`mock-exam-student-api::handleGetResult`) and the tutor review (`mock-exam-tutor-api::handleGetAttempt`) build `part1_answers`; when a Part 1 task has **no row OR `earned_score === null`** but a resolved answer exists — **`value = student_answer ?? ai_part1_ocr_json.cells[kim].value`** (typed digital-input first, else OCR-recognized blank) — **derive** `earned_score = checkPart1(correct, value, mode, max)` **for display only** (stored row always wins). The OCR-only version (2026-06-06) missed **form mode** (no OCR JSON), so a digital-input attempt showed the total but no per-task breakdown for the student and «без ответа»/«—» for the tutor; the `student_answer ?? ocr` resolve covers both. `handleGetResult` now iterates **ALL** Part-1 variant tasks (not just rows) so the student always sees the full table. `ai_part1_ocr_json` is read server-side ONLY — **never returned to the student**. Fresh attempts persist per-KIM rows on submit (`handleSubmitAttempt` form path) / OCR (`runPart1OCR`); the fallback is a safety net for stale/edge-case attempts. **`handlePart1Finalize` recomputes to match the display:** missing KIMs with a recognized OCR value → `earned_score=checkPart1(...)` (NOT `0`); **existing rows with `earned_score === null` AND `score_source != 'tutor'`** get recomputed from `student_answer ?? ocr` (so approve/total are correct for stale form attempts). Tutor manual scores (`score_source='tutor'`) are never overwritten. Only genuinely-unrecognized cells stay `0`/`finalize_default`.
- **Unified `Part1ReviewPanel` (form + blank, 2026-06-07):** ONE editable KIM-card grid for BOTH `answer_method`s (was split: `Part1SummaryCard` table for form / `Part1BlankReviewPanel` grid for blank — `Part1SummaryCard` is **deleted**). Answer per card resolves via `resolvePart1StudentAnswer` (`student_answer ?? OCR`); label is **«Распознано:»** when `fromOcr` else **«Ответ ученика:»**. **OCR is gated by `isOcrMode = attempt.answer_method !== 'form'`** (NOT `=== 'blank'`, review fix P2): blank + legacy-NULL `answer_method` use OCR (UI + the `ocrForResolve` fallback passed to `resolvePart1StudentAnswer`/counters); explicit `'form'` NEVER touches OCR (a stale OCR blob on a form attempt must not show «Распознано»/low-conf without blank context). OCR-only UI (blank photo, «AI распознал N/20» banner, «Перезапустить AI», low-confidence ring `resolved.fromOcr && confidence==='low'`) all gate on `isOcrMode`. Counters + «Балл AI: X / макс» («Ваш балл» when `score_source === 'tutor'`) + correctness background tint (emerald/amber/rose/slate) show for both. ## Часть 2 bulk photos — upload + delete (single bulk pack)

- **Model:** ONE bulk pack per attempt — `mock_exam_attempts.part2_bulk_photo_urls` (dual-format: single ref OR JSON-array), AI two-pass assigns photos→KIM. No per-task attach (removed Phase 5). Cap = `MAX_PART2_BULK_PHOTOS` (10), mirrored frontend (`StudentMockExam.tsx`) ↔ backend (`mock-exam-student-api`).
- **Append-only + CAS:** upload appends `[...existing, ref]` under a 3-retry CAS (`UPDATE … WHERE part2_bulk_photo_urls = rawCurrent`). `handleDeleteBulkPhoto` (`POST /attempts/:id/photo/delete`, body `{kind:'part2_bulk', photo_url}`) deletes **by identity, NOT index** (ChatGPT-5.5 review fix): client sends the signed `photo_url` it renders (or a raw `storage://` ref), backend normalizes to `{bucket,path}` and matches a stored ref by path. Index was unsafe — concurrent deletes shift indices, and the GET drops failed-to-sign URLs so the UI index can diverge from the stored index. Same 3-retry CAS re-matches by path each iteration, then `storage.remove()` the blob (best-effort, rule 50 order: clear DB ref first). Photo not found → 409 `PHOTO_ALREADY_REMOVED`.
- **Status drift guard (CRITICAL):** the delete CAS UPDATE MUST include `.eq("status","in_progress")` AND re-read `status` inside the retry loop. `handleSubmitAttempt` flips `status` to `submitted` WITHOUT touching `part2_bulk_photo_urls`, so a delete checked-once-before-the-loop would still CAS-match and remove a blob from a submitted attempt — and submit fires the grader (fire-and-forget) on those photos. Status drift → 409 `NOT_IN_PROGRESS`.
- **Student delete UX:** ✕ on each thumbnail must be always-visible (not hover — touch breaks, rule 80) + a confirm step; optimistic remove with restore-on-error + `toast.error`. Upload errors go to a toast too (tiny inline text was missed by pilot students).
- **Frontend single-flight on bulk mutations (2026-06-02 review fix):** delete is identity-keyed (by url), and while a delete is in flight (`deletingUrl !== null`) BOTH other deletes AND uploads are disabled — the bulk pack has one mutation at a time. Critically, restore-on-error MUST be a **functional re-add** (`setPart2BulkPhotos(prev => prev.includes(url) ? prev : [...prev, url])`), NOT a captured `snapshot` — restoring a stale snapshot would clobber a concurrent upload's signed URL that the backend already persisted.

## Часть-1 form-mode «без ответа» — root cause + deploy-топология (2026-06-08, КРИТИЧНО)

Многораундовая диагностика. Итоговая причина — **НЕ деплой, а SELECT несуществующей колонки с проглоченной ошибкой**:

- **Root cause:** `handleGetResult` (student) и `handleGetAttempt` (tutor) делали `SELECT … tutor_comment` из `mock_exam_attempt_part1_answers`, **а колонки `tutor_comment` в этой таблице НЕТ** (она на `mock_exam_attempt_part2_solutions`; schema `20260508120000` + единственная доп-колонка `score_source` в `20260516130000`). PostgREST падал «column does not exist» → код деструктурировал **только `data`** (`const { data } = …`), `error` глотался → `part1Rows = null` → `?? []` пуст → ВСЕ ответы Части 1 = «без ответа». **Бьёт и ученика, и репетитора** (оба SELECTят `tutor_comment`). **Blank-режим выживал** (ответ берётся из `ai_part1_ocr_json` на таблице attempts, не из этих строк) — отсюда «blank работает, form нет».
- **Фикс (2 раунда):** (R1) убрал `tutor_comment` из обоих read-SELECT + **логирую `error`, а не глотаю** (`const { data, error } = …; if (error) console.error(...)`) → ответы Части 1 показались. (R2) оказалось, колонки нет и для **WRITE**: `handlePart1ManualScore` (tutor-api) UPSERT'ил `tutor_comment` (поле «Комментарий ученику» в drill-down диалоге) → **«Failed to persist manual score»** (весь UPSERT ронялся → балл ТОЖЕ не сохранялся, не только коммент). Колонка `tutor_comment` добавлена миграцией **`20260608120000`** (`ADD COLUMN IF NOT EXISTS`, nullable, student-visible — ученик видит коммент в результате; table-level GRANT покрывает), и `tutor_comment` **возвращён** в оба read-SELECT → коммент сохраняется и отображается у ученика+репетитора. **Урок:** UI-поле, пишущее в колонку, которой нет в схеме = и read, и write молча падают; всегда сверять колонку с миграцией.
- **Инварианты (НЕ повторять):** (1) НИКОГДА не SELECTить колонку, не сверившись со схемой — особенно cross-table: `tutor_comment`/`tutor_score` живут на **part2_solutions**, НЕ на part1_answers. (2) НИКОГДА не глотать `error` supabase-запроса: деструктурировать `{ data, error }` + лог. «Тихий» `const { data }` прятал этот баг. (3) Симптом «blank работает / form `без ответа`» → первым делом подозревать SELECT на `mock_exam_attempt_part1_answers`.
